# 01-组织与权限 模块设计

## 1. 文档目的

本文档定义 `01-组织与权限` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围，作为模块功能、领域模型和开发任务的上层约束文档。

对应架构决策编号：D06（租户与组织分离）、D07（权限模型）、D08（身份上下文）、D09（01 领域模型优先细化）。

## 2. 设计策略

### 2.1 参考设计原则

本模块遵循“组织主数据稳定、授权规则可组合、身份上下文统一输出”的设计思路，具体原则如下：

- **先分离主数据，再叠加授权**：组织、部门、岗位、人员、账号、角色先分别建模，再通过任职关系、岗位角色关联和权限策略组装最终授权结果。
- **先稳定 RBAC 骨架，再引入 ABAC 裁剪**：资源访问由角色承载稳定授权骨架，数据访问由行级/字段级策略做细粒度裁剪。
- **身份上下文只在运行时生效**：主岗/兼岗切换是会话态，不把“当前身份”固化回人员、岗位或任职主数据。
- **租户只负责隔离，不负责组织语义**：`tenantId` 参与数据隔离和配额边界，不参与组织层级、岗位层级和授权主体推导。
- **岗位继承优先于逐人直授**：角色优先绑定岗位，人员直授角色只作为例外机制，避免授权关系失控。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| 组织模型归 `01` 持有 | 组织、部门、岗位、人员、账号、角色及其关系的主数据统一由 `01` 建模与维护 |
| 租户模型不侵入组织授权语义 | 租户隔离由 `00` 提供，`01` 只引用 `tenantId` 做隔离，不把租户当作组织层级 |
| 资源权限与数据权限分层判定 | 资源权限由 `role-resource-auth` 负责，行级/字段级权限由 `data-permission` 负责 |
| 身份上下文是唯一授权入口 | 流程、内容、门户、消息、业务应用都必须基于 `identity-context` 输出进行判定 |
| 主数据不承载会话状态 | 主岗/兼岗切换、当前岗位、当前角色集合只在运行时上下文和缓存中生效 |
| 同步与审计不改写边界 | `org-sync-audit` 负责同步落库、差异处理和审计留痕，但不改变组织主数据和授权主数据的所有权归属 |

### 2.3 一期能力收敛口径

| 能力 | 一期保留 | 一期简化 | 一期暂缓 |
|------|----------|----------|----------|
| 五维组织模型（组织/部门/岗位/人员/角色） | 是 | - | - |
| 主岗/兼岗与身份切换 | 是 | 只围绕主岗与兼岗切换，不引入更复杂的多身份并行会话模型 | - |
| 资源权限（RBAC） | 是 | 聚焦菜单、按钮、接口、页面、资源动作 | - |
| 行级数据权限 | 是 | 聚焦全量、本组织、本部门、本岗位、本人、自定义条件 | - |
| 字段级数据权限 | 是 | 聚焦查看、编辑、导出、隐藏、脱敏动作 | - |
| 外部组织同步 | 是 | 先落同步任务框架、差异处理和审计能力，具体外部源按项目接入 | - |
| 关系型细粒度授权（ReBAC / FGA） | - | - | 是 |
| 纯 ABAC 或纯 ReBAC 全域权限模型 | - | - | 是 |

## 3. 分层架构

### 3.1 主数据建模层（Master Data Layer）

负责组织、部门、岗位、人员、账号等核心主数据建模和生命周期管理。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `org-structure/` | 组织树、部门树、层级调整、状态管理、批量导入 | `org_organization`、`org_department` |
| `position-assignment/` | 岗位定义、主岗/兼岗任职关系、岗位角色继承入口 | `org_position`、`org_assignment` |
| `person-account/` | 人员档案、账号身份、主账号/多账号约束、登录与锁定状态 | `org_person`、`org_account` |

**主数据建模层规则**：

- 组织树和部门树分别建模，组织归属组织，部门归属组织并可形成部门层级。
- 岗位同时具备组织归属和部门归属，可作为身份上下文和角色继承的锚点。
- 人员档案与登录账号分层建模，人员变化不直接破坏账号模型。
- 任职关系必须支持“唯一主岗 + 多个兼岗”，且任职状态可独立失效。

### 3.2 授权与策略层（Authorization & Policy Layer）

负责角色、资源权限、行级数据权限和字段级数据权限的授权规则定义。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `role-resource-auth/` | 角色定义、岗位-角色关联、人员直授角色、资源权限策略、角色模板 | `org_role`、`org_position_role`、`org_person_role`、`org_resource_permission` |
| `data-permission/` | 行级权限策略、字段级权限策略、权限预览、权限模板与组合判定 | `org_data_permission`、`org_field_permission` |

**授权与策略层规则**：

- 角色优先通过岗位继承，人员直授角色只作为例外补充。
- 资源权限只回答“能否访问某资源动作”，不回答“能看哪些数据”。
- 行级和字段级权限必须支持组织、部门、岗位、人员、角色等不同授权主体。
- 判定顺序遵循“身份上下文 -> 资源权限 -> 行级权限 -> 字段级权限”。
- 显式拒绝优先于显式允许，更具体主体优先于更宽泛主体。
- 模板能力只服务于授权复用和实施提效，不改变角色、资源权限和数据权限的所有权边界。

### 3.3 身份上下文与会话层（Identity Context Layer）

负责基于主岗/兼岗、任职关系和角色继承结果输出统一 `IdentityContext`。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `identity-context/` | 主岗默认上下文、兼岗切换、角色集合计算、上下文刷新通知 | 无独立主数据，依赖会话、缓存和事件 |

**身份上下文与会话层规则**：

- 默认身份为主岗，用户可显式切换到兼岗。
- 切换后必须刷新菜单、待办、消息、门户卡片和数据权限上下文。
- 身份上下文是运行时结果，不回写人员、岗位、任职等主数据。
- 任职变更、岗位变更、角色绑定变化都必须触发上下文重算和缓存失效。

### 3.4 同步与治理层（Sync & Governance Layer）

负责外部组织源接入、同步任务、差异处理和审计留痕。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `org-sync-audit/` | 同步源配置、同步任务、差异对账、人工修正、组织与授权审计 | 基于同步任务、差异记录和审计记录持久化，具体建表在实现阶段细化 |

**同步与治理层规则**：

- 同步能力负责把外部源数据落为本模块主数据，不把外部系统变成本模块内部主模型。
- 同步后的异常、冲突和手工修正都必须留审计轨迹。
- 组织变更、任职变更、授权变更的治理动作都必须可回溯、可复算。

## 4. 核心领域对象关系

```text
Organization ──1:N──> Organization
Organization ──1:N──> Department
Department   ──1:N──> Department
Organization ──1:N──> Position
Department   ──1:N──> Position
Person       ──1:N──> Assignment
Assignment   ──M:1──> Position
Person       ──1:N──> Account
Position     ──M:N──> Role              (通过 PositionRole)
Person       ──M:N──> Role              (通过 PersonRole，例外直授)
Role         ──1:N──> ResourcePermission
Organization/Department/Position/Person/Role
           ──1:N──> DataPermission
Organization/Department/Position/Person/Role
           ──1:N──> FieldPermission
IdentityContext ──M:1──> Person / Account / Assignment / Position
```

领域实体详细定义见 `domain-model.md`。

### 5.3 权限判定输出边界（D07）

| 输出能力 | 负责子模块 | 调用方使用方式 |
|----------|------------|----------------|
| 资源权限判定 | `role-resource-auth/` | 判定菜单、按钮、接口、页面、资源动作是否可访问 |
| 行级数据权限 | `data-permission/` | 输出业务对象的数据范围条件 |
| 字段级数据权限 | `data-permission/` | 输出业务对象在特定场景下的字段动作矩阵 |
| 身份上下文 | `identity-context/` | 输出当前人员、岗位、组织、角色集合和生效时间 |

**边界规则**：

- 业务模块不得绕过 `identity-context` 自行推导主岗/兼岗和角色集合。
- 业务模块不得在本地重复维护一套资源权限、行级权限或字段级权限规则。
- 本模块输出的是“判定依据”和“裁剪结果”，不是业务模块的读模型聚合层。

## 6. 身份上下文与租户隔离

### 6.1 租户与组织分离原则（D06）

- `tenantId` 是系统级隔离字段，不是组织树、部门树、岗位树的一部分。
- 组织、部门、岗位、人员、角色都在租户内完成建模和授权计算，不跨租户联算。
- 权限判定时先做租户隔离，再做组织/部门/岗位/人员/角色范围裁剪。
- 前端交互不得把“租户选择”和“组织层级选择”混用为一套业务概念。

### 6.2 身份上下文输出原则（D08）

- `IdentityContext` 至少包含 `personId`、`accountId`、`tenantId`、当前组织/部门/岗位、岗位类型和继承角色集合。
- 默认上下文对应主岗，兼岗只通过显式切换进入会话上下文。
- 身份切换后，待办、消息、门户卡片、菜单树和数据权限条件必须同步刷新。
- 任职失效、岗位停用、组织停用、人员停用、账号锁定等被动变化若影响当前激活上下文，必须通过 `org.identity-context.invalidated` 统一通知下游做回退或退出处理。
- 上下文切换结果只存在于会话/缓存，不反向修改 `Assignment` 的静态事实。

## 7. 运行治理策略

### 7.1 主数据治理

- 组织、部门的层级变更必须同步维护物化路径和层级深度。
- 岗位、人员、账号的停用不直接物理删除历史关系，只改变状态并保留引用完整性。
- 批量导入、批量调岗、批量启停用必须产出统一审计记录。

### 7.2 授权治理

- 角色停用、岗位绑定变更、人员直授变更都必须触发最终权限重算。
- 权限缓存必须可按人员、岗位、角色和业务对象粒度失效。
- 所有显式拒绝、例外直授和字段脱敏规则都必须保留决策依据，支撑审计和回溯。

### 7.3 身份治理

- 主岗/兼岗切换必须在一次会话切换动作中完成上下文重算、缓存刷新和刷新通知发布。
- 身份切换失败时不得留下部分刷新结果，避免菜单、待办和数据权限口径不一致。
- 任职到期、主岗变更等后台事件触发的身份重算应支持异步补偿。
- 不允许由下游模块自行拼接“任职失效/岗位停用/账号冻结”规则推断身份是否可继续使用；被动身份失效统一以 `org.identity-context.invalidated` 为准。

### 7.4 同步治理

- 同步任务必须支持全量、增量、失败重试和人工修正入口。
- 同步冲突不得静默覆盖本模块关键主数据，必须进入差异对账和修正流程。
- 同步动作、差异处理、人工修正都必须进入统一审计。

## 9. 一期交付范围

### 9.1 一期包含

- `org-structure`：组织/部门 CRUD、层级调整、状态管理、批量导入
- `position-assignment`：岗位定义、任职关系、唯一主岗、兼岗维护、岗位角色继承入口
- `person-account`：人员档案、账号管理、密码与锁定策略接入、外部认证接入点预留
- `role-resource-auth`：角色管理、岗位关联角色、人员直授角色、RBAC 资源权限配置、角色模板
- `data-permission`：行级数据权限、字段级数据权限、权限预览、权限模板与统一决策接口
- `identity-context`：主岗默认上下文、兼岗切换、菜单/待办/消息刷新联动
- `org-sync-audit`：同步任务框架、差异对账、异常补偿、组织与授权审计

### 9.2 一期简化

- 资源权限先聚焦菜单、按钮、接口、页面、资源动作，不扩展更复杂的细粒度关系权限。
- 行级数据权限先覆盖全量、本组织、本部门、本岗位、本人和条件表达式，不引入更复杂关系推导。
- 字段级权限先覆盖查看、编辑、导出、隐藏、脱敏动作，不引入独立字段工作流。
- 身份切换先围绕主岗/兼岗，不引入多身份并行激活或跨端独立身份栈。
- 外部组织同步先建立统一接入框架、差异治理和审计能力，具体外部源按项目落地。
- 角色模板和权限模板先聚焦定义、复用和导入导出，不做更复杂的市场化分发或跨项目治理。

### 9.3 一期暂缓

- 共享协作对象的 ReBAC / Fine-Grained Authorization 模型
- 纯 ABAC 或纯 ReBAC 替代 RBAC 骨架的全域权限体系
- 跨租户组织模型和跨租户授权计算

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 租户模型与组织模型混用，导致授权口径紊乱 | 明确 `tenantId` 只做隔离，组织/岗位/角色语义统一留在 `01` |
| 逐人直授角色过多，导致授权不可维护 | 强制岗位继承优先，人员直授只作为例外并要求审计原因 |
| 资源权限、行级权限、字段级权限冲突难以解释 | 固化判定顺序与显式拒绝优先规则，输出可审计决策依据 |
| 身份切换后缓存未同步失效，出现菜单和数据口径不一致 | 统一通过 `identity-context` 触发重算、失效和刷新通知 |
| 外部同步冲突破坏主数据稳定性 | 同步先入差异治理流程，人工修正与补偿动作全量审计 |
