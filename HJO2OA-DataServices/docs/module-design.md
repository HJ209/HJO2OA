# 07-数据服务与集成 模块设计

## 1. 文档目的

本文档定义 `07-数据服务与集成` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和分阶段交付范围。作为模块功能文档、领域模型文档和开发任务文档的上层约束文档。

对应架构决策编号：D05（`04` 与 `07` 数据边界）、D12（统一事件契约）、D13（最终一致性）、D17（数据开放分层）、D20（一期实施范围）。

## 2. 设计策略

### 2.1 参考设计原则

本模块参考 O2OA 的数据中心、服务管理和开放能力思路，但不沿用“门户聚合、外部开放、连接器配置和报表能力混为一层”的做法。具体原则如下：

- **借鉴平台化开放能力**：统一数据服务、开放 API、连接器、同步、报表和治理能力的收口方式。
- **不借鉴用户入口与开放入口混用**：门户首页和办公中心聚合仍归 `04`，`07` 面向通用开放、跨系统复用和治理。
- **不借鉴业务域各自暴露接口**：业务域应先输出稳定内部协议或事件，再由 `07` 统一装配为对外能力。
- **优先定义契约和边界，再做运行编排**：先收敛服务定义、版本、鉴权、映射、调度和告警规则，再决定具体执行方式。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| `07` 拥有开放和集成真相源 | 服务定义、开放接口、连接器、同步任务、统计口径、治理策略归 `07` |
| 用户侧聚合读模型不归 `07` | 门户首页和办公中心聚合接口归 `04`，`07` 只提供可复用数据源 |
| 消息触达不归 `07` | 触达、渠道、站内信和移动消息归 `06` |
| 业务主数据不归 `07` | `07` 只引用 `01/02/03/05/06` 的业务数据或事件，不拥有主表 |
| 同步和报表是投影 | 同步状态、对账结果、报表快照不能替代源系统或业务系统真相源 |
| 开放能力必须可治理 | 版本、凭证、配额、限流、审计、告警、补偿都必须统一建模 |

### 2.3 能力借鉴清单

| 平台能力 | HJO2OA 一期保留 | HJO2OA 一期简化 | HJO2OA 暂缓 |
|----------|----------------|----------------|-------------|
| 数据服务定义 | - | 建模与占位 | 大规模编排 |
| 开放 API 目录与鉴权 | - | 建模与标准协议 | 复杂开发者门户 |
| 连接器框架 | - | HTTP/DB/MQ 三类优先 | 全量 SaaS 连接器市场 |
| 同步任务 | - | 基础全量/增量/事件驱动模型 | 实时 CDC |
| 报表与统计 | - | 基础指标、趋势、图卡数据源 | 高级 BI |
| 服务治理与告警 | - | 健康检查、告警规则、版本治理 | 智能诊断/自动修复 |

## 3. 分层架构

### 3.1 服务定义层（Service Definition Layer）

负责内部可复用数据服务、开放接口协议和连接器配置的定义真相源。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `data-service/` | 服务定义、参数模型、字段映射、权限边界、缓存策略 | `data_service_def`、`data_service_param_def`、`data_service_field_mapping` |
| `open-api/` | 对外接口目录、版本、鉴权、凭证、限流、配额 | `data_open_api_endpoint`、`data_api_credential_grant`、`data_api_rate_limit_policy` |
| `connector/` | 连接器类型、参数、健康检查和测试能力 | `data_connector_def`、`data_connector_param`、`data_connector_health_snapshot` |

**服务定义层规则**：

- 所有对外能力必须先定义服务协议，再装配为开放接口或同步任务。
- 对外 API 优先复用数据服务定义，减少重复实现和协议漂移。
- 敏感连接参数必须走 `00-security` 密钥引用，不允许明文散落。

### 3.2 集成执行层（Integration Execution Layer）

负责跨系统同步、事件驱动交换、检查点和补偿。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `data-sync/` | 同步任务、源目标映射、检查点、失败重试、补偿与对账 | `data_sync_task`、`data_sync_mapping_rule`、`data_sync_execution_record` |

**集成执行层规则**：

- 同步任务的真相源是任务定义和映射规则，执行记录是运行态投影。
- 增量、事件驱动和全量同步必须显式声明检查点与冲突策略。
- 同步任务不应直接修改业务模块定义边界外的数据表，写入必须通过目标系统或目标模块的稳定协议。

### 3.3 分析输出层（Analytics Output Layer）

负责报表口径、统计快照和图卡数据源输出。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `report/` | 指标定义、维度建模、聚合查询、趋势排行、图卡输出协议 | `data_report_def`、`data_report_metric_def`、`data_report_snapshot` |

**分析输出层规则**：

- 报表定义拥有统计口径真相源，门户图卡只消费其输出。
- 报表快照可由调度或事件刷新，但必须记录新鲜度与刷新时间。
- 统计输出必须支持按租户、组织或权限范围裁剪。

### 3.4 治理运行层（Governance Runtime Layer）

负责开放能力的运行治理、健康检查、版本与告警。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `governance/` | 版本治理、健康检查、告警规则、异常链路追踪、人工补偿入口 | `data_governance_profile`、`data_health_check_rule`、`data_alert_rule`、`data_service_version_record` |

**治理运行层规则**：

- 治理动作只能通过显式策略和治理入口触发，不得绕过服务定义与同步定义直接改底层数据。
- 告警、重试、停用、回滚和补偿都必须进入统一审计。
- 治理层优先围绕开放能力本身，不替代业务域的业务监控。

## 5. 跨模块集成

### 5.1 依赖关系

| 依赖方向 | 依赖内容 | 集成方式 |
|----------|----------|----------|
| `07 -> 00` | 事件总线 | 通过统一事件总线消费业务事件、发布同步和治理事件 |
| `07 -> 00` | 调度与配置 | 通过 `scheduler` 触发同步和报表任务，通过 `config` 管理连接参数与策略覆盖 |
| `07 -> 00` | 安全与审计 | 通过 `security` 管理密钥引用、签名与敏感配置，通过 `audit` 记录治理动作 |
| `07 -> 01` | 身份上下文与数据权限 | 按调用方身份、组织维度和授权范围裁剪开放结果 |
| `07 -> 02` | 流程与待办数据 | 统计、开放接口与同步的数据来源 |
| `07 -> 03` | 内容与知识数据 | 内容开放、订阅同步与报表来源 |
| `07 -> 04` | 报表图卡输出协议 | `04` 消费 `07-report` 的图卡数据源，不反向拥有开放能力 |
| `07 -> 05` | 会议、任务、考勤、合同等业务数据 | 业务开放、同步与报表来源 |
| `07 -> 06` | 告警触达与外部通知 | 治理事件交给 `06` 做消息分发与移动端触达 |

### 5.2 事件契约

本模块发布的领域事件遵循 `docs/contracts/unified-event-contract.md`，模块前缀为 `data`。

| 事件类型 | 触发时机 | 消费模块 |
|----------|----------|----------|
| `data.service.activated` | 数据服务启用 | 00/06/07 |
| `data.api.published` | 开放接口发布 | 06/07 |
| `data.api.deprecated` | 开放接口废弃 | 06/07 |
| `data.connector.updated` | 连接器配置变更 | 07 |
| `data.sync.completed` | 同步任务完成 | 00/06/07 |
| `data.sync.failed` | 同步任务失败 | 00/06/07 |
| `data.report.refreshed` | 报表刷新完成 | 04/07 |
| `data.governance.alerted` | 治理告警产生 | 06/07 |

### 5.3 与最终一致性机制的关系（D13）

| 场景 | `07` 提供的能力 | 业务模块职责 |
|------|----------------|--------------|
| 外部系统回写 | 定义开放接口和幂等策略 | 保证业务语义和目标资源校验 |
| 跨系统同步 | 提供检查点、对账、补偿、告警 | 保证源数据和目标数据规则正确 |
| 报表刷新 | 提供统计口径与快照刷新机制 | 保证上游事件或查询语义稳定 |
| 治理告警 | 提供健康检查、阈值和追踪 | 保证业务模块输出必要状态与错误上下文 |

### 5.4 与用户入口和消息触达的边界

| 能力 | 归属 | 说明 |
|------|------|------|
| 门户首页和办公中心聚合 | `04` | 面向用户入口体验 |
| 统一消息中心和渠道触达 | `06` | 面向用户通知与移动入口 |
| 通用开放接口、连接器、同步、报表和治理 | `07` | 面向第三方系统和平台运营 |

## 6. 数据所有权

| 数据 | 所有者 | 说明 |
|------|--------|------|
| 数据服务定义、参数协议、字段映射 | `07 data-service` | 平台内部与对外服务定义真相源 |
| 开放接口、凭证、限流与配额 | `07 open-api` | 平台开放边界真相源 |
| 连接器定义、参数、健康快照 | `07 connector` | 外部资源访问能力真相源 |
| 同步任务、检查点、执行记录、对账差异 | `07 data-sync` | 跨系统交换运行真相源 |
| 报表定义、指标、快照 | `07 report` | 平台统计口径与输出真相源 |
| 治理策略、健康检查、告警、版本记录 | `07 governance` | 开放能力治理真相源 |
| 业务主数据 | 各业务模块 | `07` 只引用，不拥有 |
| 门户聚合页级读模型 | `04` | `07` 只提供图卡数据源 |
| 消息触达记录与渠道状态 | `06` | `07` 通过事件协同，不直接拥有 |

## 7. 分阶段交付范围

### 7.1 一期包含

- 完成 `07` 模块边界、领域模型和工程骨架预留
- 收敛开放接口、连接器、同步、报表和治理的统一文档口径
- 为后续 `bootstrap-with-phase2-reserved` Profile 接入预留模块装配边界

### 7.2 一期简化

- 不实现完整开发者门户、连接器市场和高级可视化编排
- 不实现复杂实时同步、流式处理和高级 BI
- 只保留标准数据服务定义、接口版本、连接器参数、同步任务和统计口径模型

### 7.3 二期/后续优先实现

- `data-service`：统一查询/提交服务定义与内部复用
- `open-api`：接口目录、鉴权、配额和调用审计
- `connector`：HTTP/数据库/MQ 三类基础连接器
- `data-sync`：同步任务、检查点、补偿和对账
- `report`：门户图卡和管理分析页所需基础报表
- `governance`：健康检查、版本治理和异常告警

## 8. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| `07` 演变成“万能中间层” | 强制 `07` 只拥有开放与集成真相源，不反向侵入业务主数据 |
| `04` 与 `07` 边界再次混淆 | 明确 `04` 做用户入口聚合，`07` 做通用开放与治理 |
| 连接器配置失控带来安全风险 | 全量密钥引用、审计留痕、权限控制和环境隔离 |
| 同步任务失败后形成脏数据 | 检查点、对账、补偿、告警和人工介入入口共同保障 |
| 报表口径漂移导致数据不可信 | 统计口径版本化、快照时间标记和审计记录 |
| 对外接口变更影响调用方 | 版本治理、兼容策略、废弃公告和灰度下线 |

## 10. 结论

`07-数据服务与集成` 的关键不是把所有数据访问都集中到一个大模块，而是把**通用开放、跨系统连接、同步交换、统计输出和运行治理**收敛成一套稳定边界。其设计重点应围绕以下原则展开：

- 用 `data-service` 统一内部可复用服务协议
- 用 `open-api` 统一对外开放入口和版本治理
- 用 `connector` 统一访问外部资源的方式
- 用 `data-sync` 统一跨系统数据交换和补偿
- 用 `report` 统一统计口径与图卡数据源
- 用 `governance` 统一开放能力运行治理与异常闭环

该文档可直接作为后续数据库设计、接口契约细化、连接器实现和二期工程接入的上层约束。
