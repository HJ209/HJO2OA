# process-instance 概览

## 模块定位

`process-instance` 是 `02-流程与表单` 的运行时主模块，负责 `ProcessInstance` 和 `TaskInstance` 生命周期。它承接流程发起、实例状态推进、任务创建与认领、轨迹查询、草稿恢复和实例归档，是流程运行态的权威写模型。

## 核心职责

- 基于已发布流程定义和表单元数据创建流程实例，并冻结定义/表单快照。
- 管理实例状态 `RUNNING / COMPLETED / TERMINATED / SUSPENDED` 与任务状态变化。
- 提供草稿保存、草稿恢复、草稿提交能力。
- 提供流程详情、节点轨迹、实例控制和归档数据输出。
- 在关键状态变化后发布 `process.instance.*` 与 `process.task.*` 事件，供待办、消息和监控链路消费。

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `process-definition` | 只能使用已发布流程定义，实例创建时冻结定义版本快照。 |
| 上游 | `form-metadata` / `form-renderer` | 只持有表单元数据快照和 `formDataId` 引用，不拥有业务表单正文。 |
| 上游 | `05-协同办公应用` | 业务域提供流程标题、业务单据 ID、初始表单数据，实例模块不承载业务语义。 |
| 协同 | `action-engine` | 实例模块拥有生命周期，动作执行校验和动作留痕由 `action-engine` 负责。 |
| 下游 | `todo-center` / `06-消息移动与生态` / `process-monitor` | 通过任务和实例事件驱动待办投影、消息触达和后续监控。 |

## 核心能力表

| 能力 | 输入 | 输出 | 关键约束 |
|------|------|------|----------|
| 流程发起 | 定义版本、`formDataId`、标题、发起人身份上下文 | `ProcessInstance`、首批 `TaskInstance` | 仅已发布定义可发起。 |
| 任务管理 | 候选人、认领请求、动作执行结果 | 任务状态与当前节点集合 | 任务状态推进由动作结果驱动。 |
| 草稿管理 | 未提交表单输入、定义选择 | 草稿快照 | 草稿不产生领域事件。 |
| 实例控制 | 挂起、恢复、终止 | 实例状态变更 | 挂起后暂停超时计时。 |
| 归档输出 | 已结束实例 | 归档明细和查询入口 | 归档只发生在完成/终止之后。 |

## 典型业务流

1. 业务应用选择已发布流程定义，调用发起接口保存草稿或直接提交。
2. 提交时冻结定义版本、表单元数据和发起人身份上下文，创建实例与首节点任务。
3. 审批页面通过 `action-engine` 执行动作，实例模块根据执行结果推进节点和任务。
4. 实例结束后关闭未完成任务，生成归档可查询数据，并向下游发布完成或终止事件。
5. 待办、消息、后续监控均通过事件异步更新，不反向改写实例主表。

## 一期实现约束

- 只支持基础发起、任务认领、流程挂起/恢复/终止、轨迹查询和归档输出。
- 业务表单数据仍归业务域所有，实例模块只保存 `formDataId` 与必要快照。
- 老实例始终沿用启动时的定义版本和表单版本，不受后续发布影响。
- 草稿仅用于发起前或未完成提交前的恢复，不替代业务域正式数据保存。
- 移动端只复用发起/处理壳层页面，不提供流程管理后台能力。
