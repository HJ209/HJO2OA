# 02-流程与表单 模块设计

## 1. 文档目的

本文档定义 `02-流程与表单` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围。作为模块功能、领域模型和开发任务的上层约束文档。

对应架构决策编号：D10（02 领域模型优先细化）、D16（待办与消息分工）、ADR-001（Flowable）、ADR-006（参考设计策略）。

## 2. 设计策略

### 2.1 参考设计原则

本模块参考 O2OA 的能力模型与交互闭环，但不沿用其产品结构与源码组织方式（ADR-006）。具体原则：

- **借鉴能力闭环**：流程定义/实例、表单元数据、统一待办、审计留痕等成熟设计经验直接复用思路。
- **不借鉴产品结构**：O2OA 的流程/表单/门户/数据服务天然更紧，HJO2OA 需保持清晰模块边界。
- **不借鉴实现方式**：身份解析、权限判定、消息触达等关键边界必须走 HJO2OA 统一契约，不在流程域内重复实现。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| 流程引擎只负责状态推进与动作编排 | 不承载业务语义、不存储业务主数据 |
| 表单引擎只负责结构描述、展示规则、输入采集 | 不承载业务规则、不直接操作业务数据 |
| 业务主数据归业务域所有 | 流程域仅维护流程状态、任务、动作和表单快照/引用 |
| 身份与角色解析由统一身份上下文提供 | 流程域不自行实现组织权限判断逻辑 |
| 字段级/行级权限走统一权限模型 | 不在流程域内重复造权限体系 |
| 待办是事务处理视图，消息是通知触达视图 | 两者联动但不混同（D16） |

### 2.3 O2OA 能力借鉴清单

| O2OA 能力 | HJO2OA 一期保留 | HJO2OA 一期简化 | HJO2OA 暂缓 |
|-----------|----------------|----------------|-------------|
| 流程定义版本化 | ✅ | - | - |
| 流程实例/任务模型 | ✅ | - | - |
| 节点动作（通过/驳回/退回/转办/加签/撤回） | ✅ | - | - |
| 审批意见与附件 | ✅ | - | - |
| 节点级字段状态控制 | ✅ | - | - |
| 统一待办/已办/我发起/抄送 | ✅ | - | - |
| 全链路审计留痕 | ✅ | - | - |
| 流程版本冻结（新实例走新版，老实例走旧版） | ✅ | - | - |
| 复杂脚本引擎 | - | 简化为条件表达式 | - |
| 重低代码表达式 | - | 简化为联动规则 | - |

## 3. 分层架构

### 3.1 设计时（Design-Time）

负责流程定义与表单元数据的建模、版本管理和发布。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `process-definition/` | 流程模型 CRUD、节点/路由配置、参与者规则、动作配置、版本发布/废弃 | `proc_definition`、`proc_action_def` |
| `form-metadata/` | 表单字段/布局/校验定义、节点字段权限映射、版本发布/废弃 | `form_metadata` |

**设计时规则**：

- 流程定义和表单元数据均采用版本链管理，发布后不可修改，修改需创建新版本。
- 流程定义发布后才能发起实例，已发布版本可被新版本替代（DEPRECATED）。
- 起始节点有且仅有一个，结束节点至少一个。
- USER_TASK 节点必须配置参与者规则。

### 3.2 运行时（Run-Time）

负责流程实例的创建、任务流转、动作执行和状态推进。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `process-instance/` | 流程发起、实例状态管理、挂起/恢复/终止、归档 | `proc_instance` |
| `action-engine/` | 统一动作抽象、动作规则判定、动态参与者计算、条件路由 | `proc_task_action`（动作记录） |

**运行时规则**：

- 流程实例创建时冻结当前定义版本和表单元数据快照。
- 发起人身份上下文（组织、部门、岗位）在发起时快照，后续身份切换不影响已发起流程。
- 任务创建时根据参与者规则计算候选人，候选人可认领（CLAIMED）。
- 会签任务需多人处理，按完成条件判定是否流转；或签任务任一人处理即流转。
- 每次任务处理必须记录动作，动作记录不可修改和删除，作为审计依据。
- 流程完成或终止后，所有未完成任务自动取消。

### 3.3 读模型（Query Model）

负责面向用户的查询、聚合和展示。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `todo-center/` | 待办/已办/我发起/抄送/草稿/归档聚合查询、筛选、分页、统计 | `todo_item`（投影表） |
| `process-monitor/` | 流程耗时分析、停滞分析、超时预警、催办 | 读模型查询，无独立持久化 |

**读模型规则**：

- TodoItem 从 TaskInstance 投影而来，不独立创建。
- 任务完成/终止后，对应待办项状态同步更新。
- 待办事件发布后，由 `06-消息移动与生态` 消费并决定触达渠道。
- 身份上下文切换后，待办列表和待办数量必须同步刷新（D08）。

### 3.4 前端渲染层

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `form-renderer/` | 根据表单元数据协议动态渲染、字段联动、多态渲染（创建/编辑/查看/审批态） | 无独立后端持久化 |

**渲染层规则**：

- 渲染协议由 `form-metadata` 定义，渲染器只消费协议输出。
- 字段权限映射以 `{nodeId -> {fieldCode -> {visible, editable, required}}}` 格式，渲染器根据当前节点和身份上下文决定字段状态。
- 联动规则由表单元数据驱动，渲染器执行，不发布领域事件。

## 4. 核心领域对象关系

```
ProcessDefinition ──1:N──> ProcessDefinition     (版本链)
ProcessDefinition ──1:N──> ProcessInstance       (定义 -> 实例)
ProcessDefinition ──1:N──> ActionDefinition      (定义 -> 动作配置)
ProcessDefinition ──1:1──> FormMetadata          (定义 -> 表单绑定)
ProcessInstance   ──1:N──> TaskInstance          (实例 -> 任务)
ProcessInstance   ──1:1──> FormMetadata           (实例 -> 表单快照)
TaskInstance      ──1:N──> TaskAction            (任务 -> 动作记录)
TaskInstance      ──1:1──> TodoItem              (任务 -> 待办项)
```

领域实体详细定义见 `领域模型.md`。

### 5.3 待办与消息分工（D16）

| 视图 | 职责 | 管理模块 |
|------|------|----------|
| 待办视图 | 事务处理入口，待办/已办/我发起/抄送/草稿聚合与流转 | `02-流程与表单` |
| 消息视图 | 通知触达入口，站内信/邮件/短信/企业微信/钉钉多渠道路由 | `06-消息移动与生态` |

映射规则：

- 待办创建 → 消息提醒（站内信 + 可选外部渠道）
- 待办超时 → 催办提醒（站内信 + 外部渠道升级）
- 待办完成 → 不产生消息（事务闭环）
- 抄送到达 → 阅知提醒（站内信）

## 6. 身份上下文集成（D08）

| 阶段 | 集成方式 |
|------|----------|
| 流程发起时 | 发起人身份上下文（组织、部门、岗位）快照到 ProcessInstance |
| 任务分配时 | 基于 ParticipantRule + `01-组织与权限` 身份上下文计算候选人 |
| 任务处理时 | 处理人身份上下文快照到 TaskAction，权限判定基于当前身份 |
| 身份切换后 | 待办列表、待办数量、门户待办卡片同步刷新；已分配任务不重新分配 |

参与者规则支持的类型：INITIATOR / ORG_MANAGER / DEPT_MANAGER / POSITION_HOLDER / ROLE_HOLDER / SPECIFIC_PERSON / FORM_FIELD_VALUE / EXPRESSION。

## 7. Flowable 集成策略（ADR-001）

### 7.1 封装原则

- 封装 Flowable API 为平台统一接口，不直接暴露 Flowable 原生 API。
- 流程定义使用 BPMN 2.0 XML 格式，由设计器生成，Flowable 负责解析和执行。
- 流程定义的参与者计算需自定义实现 TaskListener，与 `01-组织与权限` 的身份上下文集成。

### 7.2 核心封装层

| 封装内容 | 说明 |
|----------|------|
| 流程定义服务 | 封装 RepositoryService，提供定义 CRUD、版本发布/废弃 |
| 流程实例服务 | 封装 RuntimeService，提供发起、挂起、恢复、终止 |
| 任务服务 | 封装 TaskService，提供查询、认领、转办、委派 |
| 动作执行服务 | 封装 TaskService.complete + TaskListener，提供统一动作入口 |
| 参与者解析器 | 自定义 TaskListener，调用 identity-context 计算候选人 |
| 超时监听器 | 自定义 BoundaryEvent + Timer，发布 `process.task.overdue` 事件 |

## 9. 一期交付范围

### 9.1 一期包含

- `process-definition`：流程定义 CRUD、节点/路由配置、参与者规则、版本发布/废弃
- `process-instance`：流程发起、任务流转、审批轨迹、归档
- `action-engine`：统一动作抽象（通过/驳回/退回/转办/加签/撤回/催办）、规则判定、动态参与者
- `form-metadata`：表单字段/布局/校验定义、节点字段权限映射、版本管理
- `form-renderer`：基础字段渲染、子表、字段联动、多态渲染（创建/编辑/查看/审批态）
- `todo-center`：待办/已办/我发起/抄送/草稿聚合查询

### 9.2 一期简化

- 条件路由：支持简单表达式和表单字段值判断，不支持复杂脚本
- 联动规则：支持显隐/必填/默认值联动，不支持跨表单级联脚本
- 设计器：提供基础流程设计器和表单设计器，不含完整低代码能力

### 9.3 一期暂缓

- `process-monitor`：流程监控与分析（耗时分析、停滞分析、超时预警）
- 重脚本低代码引擎
- 复杂门户深度联动
- 泛化业务数据中心
- 移动端表单渲染器
- 完整在线设计器高级能力

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| Flowable 框架较重，学习曲线陡峭 | 封装统一 API 层，团队仅接触平台接口 |
| 参与者计算与组织权限耦合复杂 | 参与者解析器独立为可测试组件，与 identity-context 通过接口解耦 |
| 表单元数据版本管理复杂度高 | 一期简化联动规则，降低元数据复杂度 |
| 待办投影一致性 | 任务事件驱动投影更新，消费端幂等 |
| 业务数据与流程数据边界模糊 | 强制 formDataId 引用模式，流程域不直接操作业务表 |
