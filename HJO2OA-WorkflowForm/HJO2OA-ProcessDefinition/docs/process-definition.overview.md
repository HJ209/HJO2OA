# process-definition 概览

## 模块定位

`process-definition` 是 `02-流程与表单` 的设计时子模块，负责维护 `ProcessDefinition` 聚合及其版本链。它输出可发布的流程模型，供 `process-instance` 在发起时冻结快照，供 `action-engine` 在运行时读取节点动作、路由和参与者规则。

## 核心职责

- 维护流程定义草稿、发布版、废弃版三种状态。
- 维护节点、路由、参与者规则、节点可用动作编码和表单绑定关系。
- 在发布前执行结构校验，阻止无起止节点、无参与者规则、路由不闭合等非法模型进入运行时。
- 管理同一 `code` 下的版本链，新实例永远只使用显式发布的版本。

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `form-metadata` | 只能绑定已发布的表单版本；本模块不维护表单字段、布局和联动规则。 |
| 上游 | `01-组织与权限` | 仅复用身份上下文、组织树、岗位树、角色集合做参与者规则预览，不自行实现组织权限逻辑。 |
| 下游 | `process-instance` | 只输出发布后的定义快照；不负责流程发起、任务创建、实例状态流转。 |
| 下游 | `action-engine` | 只定义节点可用动作编码和路由目标约束；不负责动作执行和审批留痕。 |
| 边界外 | 业务主数据 | 不存储业务单据正文，仅持有 `formMetadataId` 和流程配置。 |

## 核心能力表

| 能力 | 输入 | 输出 | 关键约束 |
|------|------|------|----------|
| 流程定义 CRUD | 名称、编码、分类、表单绑定 | 草稿版定义 | `code + version` 租户内唯一。 |
| 节点与路由建模 | 节点类型、路由条件、布局坐标 | `nodes`、`routes` JSON | 仅支持简单表达式和字段值判断。 |
| 参与者规则配置 | 组织/岗位/角色/表单字段/表达式 | `participantRule` | `USER_TASK` 节点必须有参与者规则。 |
| 节点动作配置 | `actionCodes`、目标节点约束 | 运行时动作白名单 | 只配置动作，不定义动作执行结果。 |
| 版本发布/废弃 | 草稿定义、校验结果 | `PUBLISHED` / `DEPRECATED` 版本 | 已发布版本不可原地修改。 |

## 典型业务流

1. 设计人员新建流程定义草稿，填写 `code`、名称、分类并绑定已发布表单版本。
2. 在设计器中配置起始节点、审批节点、网关路由、参与者规则和节点动作。
3. 调用校验接口检查起止节点、路由闭合、动作配置和表单绑定是否合法。
4. 校验通过后发布当前草稿；`process-instance` 只能基于发布版发起实例。
5. 后续变更通过“基于当前版创建新版本”完成，旧版保留给已运行实例继续使用，需要时显式废弃。

## 一期实现约束

- 仅提供基础流程设计器，不建设完整低代码脚本引擎。
- 条件路由只支持简单表达式、表单字段值判断和默认路由，不支持自定义脚本。
- 已发布版本不可编辑；如需修改，必须创建新版本并重新发布。
- 不提供“覆盖式回滚”，历史版本只支持对比和重新派生新版本。
- 仅管理端可编辑流程定义，移动端不提供设计能力。
