# 最终一致性与补偿机制文档

对应架构决策：D13（最终一致性）。

## 1. 文档目的

定义 HJO2OA 跨模块异步链路的最终一致性保障机制，包括重试策略、死信处理、补偿机制、对账策略和审计要求。所有跨模块事件消费链路必须遵循本文档。

## 2. 一致性模型

### 2.1 核心原则

- 跨模块业务联动通过事件总线异步完成，不追求强一致性。
- 每个事件消费者独立保证自身数据的一致性。
- 通过重试 + 死信 + 补偿 + 对账的组合策略，确保系统在有限时间内达到最终一致。

### 2.2 一致性窗口

| 场景 | 可接受延迟 | 说明 |
|------|-----------|------|
| 权限重算 | < 5 秒 | 角色变更后权限应快速生效 |
| 待办刷新 | < 3 秒 | 任务创建后待办应立即可见 |
| 消息通知 | < 10 秒 | 事件触发后消息应在短时间内送达 |
| 组织同步 | < 30 分钟 | 外部系统数据同步可容忍较长延迟 |
| 搜索索引更新 | < 60 秒 | 内容发布后搜索应可检索到 |

## 3. 事件发布保障（Outbox 模式）

### 3.1 Outbox 表设计

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 记录 ID |
| eventId | UUID | 事件唯一标识 |
| eventType | String | 事件类型 |
| eventVersion | String | 事件版本 |
| aggregateType | String | 聚合类型 |
| aggregateId | UUID | 聚合 ID |
| payload | JSON | 事件载荷 |
| status | Enum | 状态：PENDING / PUBLISHED / FAILED |
| publishedAt | Timestamp | 发布时间 |
| retryCount | Integer | 重试次数 |
| createdAt | Timestamp | 创建时间 |

### 3.2 发布流程

1. 业务操作与 Outbox 记录在**同一数据库事务**中写入。
2. 定时任务（每 100ms）扫描 `status=PENDING` 的记录，批量发布到 RabbitMQ。
3. 发布成功后标记 `status=PUBLISHED`。
4. 发布失败时递增 `retryCount`，超过阈值后标记 `status=FAILED` 并告警。

### 3.3 幂等保障

- 消费者通过 `eventId` 去重，同一事件不重复处理。
- RabbitMQ 消息设置 `messageId = eventId`，便于消费者结合幂等表或缓存进行去重处理。

## 4. 消费端重试策略

### 4.1 重试配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大重试次数 | 5 | 超过后进入死信队列 |
| 重试间隔 | 1s, 5s, 30s, 5m, 30m | 指数退避 |
| 重试队列 TTL | 1 小时 | 超时未消费则进入死信 |
| 并发消费线程数 | 按队列配置 | 默认 10 |

### 4.2 重试实现

- 消费者业务处理失败时 NACK（`requeue=false`）。
- RabbitMQ 通过 DLX（Dead Letter Exchange）路由到重试队列。
- 重试队列设置 TTL，到期后重新路由到原队列。
- 超过最大重试次数后路由到死信队列。

## 5. 死信队列处理

### 5.1 死信分类

| 分类 | 处理方式 |
|------|----------|
| 载荷反序列化失败 | 人工排查，修复后重新发布 |
| 业务逻辑不可恢复错误 | 人工排查，修正数据后补偿 |
| 版本不兼容 | 升级消费者后重新消费 |
| 超过重试上限 | 人工排查，判断是否可补偿 |

### 5.2 死信管理

- 死信队列消息持久化，不自动丢弃。
- 提供管理界面查看死信消息内容和错误原因。
- 支持手动重发（修正后重新投递到原队列）。
- 支持手动丢弃（确认无需处理后删除）。
- 死信队列消息量超过阈值时自动告警。

## 6. 补偿机制

### 6.1 补偿策略

| 策略 | 适用场景 | 说明 |
|------|----------|------|
| 自动重试 | 临时性故障（网络抖动、锁冲突） | 消费端自动重试，无需人工介入 |
| 事件补偿 | 业务逻辑部分成功部分失败 | 发布补偿事件，消费者自行修正 |
| 手动补偿 | 数据不一致需人工判断 | 管理员通过界面执行补偿操作 |
| 定时对账 | 长期运行后的数据漂移 | 定时任务比对源端与消费端数据 |

### 6.2 补偿事件

当消费者处理部分成功但后续步骤失败时，发布补偿事件回滚已完成的操作：

```
org.organization.created → 补偿事件：org.organization.creation-revoked
org.role.person-granted → 补偿事件：org.role.person-grant-revoked
```

补偿事件在事件注册表中标注 `compensatable: true`，消费者需实现对应的补偿处理逻辑。

### 6.3 手动补偿操作

| 操作 | 说明 |
|------|------|
| 重新发布事件 | 从 Outbox 表重新发布指定事件 |
| 重新消费死信 | 将死信消息重新投递到原队列 |
| 数据修正 | 直接修正消费端不一致数据 |
| 全量对账 | 触发源端与消费端的数据比对 |

## 7. 对账策略

### 7.1 定时对账

| 对账场景 | 频率 | 比对方式 |
|----------|------|----------|
| 组织权限一致性 | 每日 | 比对 01 模块的角色集合与 02/04 模块的权限缓存 |
| 待办与任务一致性 | 每小时 | 比对 02 模块的 TaskInstance 与 `todo-center` 投影数据 |
| 消息投递一致性 | 每日 | 比对事件发送记录与消息送达记录 |
| 外部同步一致性 | 每次同步后 | 比对源端与目标端数据差异 |

### 7.2 对账结果处理

- 一致：记录对账通过日志。
- 不一致且可自动修复：执行补偿事件，记录补偿日志。
- 不一致且不可自动修复：生成异常工单，人工处理。

## 8. 审计要求

- 所有事件发布和消费必须记录审计日志。
- 审计日志包含：eventId、eventType、operatorAccountId、operatorPersonId、tenantId、处理结果、耗时。
- 补偿操作必须记录补偿原因、补偿方式和补偿结果。
- 死信处理必须记录处理人、处理方式和处理结果。
- 审计日志保留 90 天，关键操作日志保留 1 年。

## 9. 关键链路示例

### 9.1 角色变更 → 权限重算

```
01 角色变更 → 发布 org.role.position-bound
→ 01 消费：重算角色集合，标记权限缓存失效
→ 02 消费：重算流程任务候选人
→ 04 消费：重算门户卡片可见范围
```

失败处理：重试 5 次后进死信 → 管理员手动补偿 → 对账确认。

### 9.2 流程任务创建 → 待办创建 → 消息通知

```
02 任务创建 → 更新 `todo-center` 投影并发布 todo.item.created
→ 04 消费：刷新办公中心待办列表与聚合卡片
→ 06 消费：发送待办提醒通知
```

失败处理：06 消费失败时重试 → 死信 → 手动补偿通知发送记录 → 对账确认。
