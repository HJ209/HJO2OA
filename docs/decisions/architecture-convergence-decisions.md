在进入编码前，优先补齐领域模型、接口契约、事件契约和一期实施切片文档，再进入工程搭建。

## 4. P02 补充判断：微服务演进策略

## 4.1 当前不立即拆微服务

当前阶段不建议直接拆为全面微服务，原因如下：

- 组织权限、流程表单、门户聚合等核心能力尚未完成领域模型和接口契约收敛
- 过早拆分会把复杂度从“业务建模”转移到“分布式治理”
- 当前仓库仍处于设计阶段，更适合先完成统一底座与最小闭环
- 平台从 0 到 1 时，模块化单体更利于快速交付、统一事务边界和统一权限治理

## 4.2 当前建议架构形态

建议采用以下形态：

- 主业务系统采用模块化单体
- 网关、对象存储、缓存、消息队列、搜索、文件预览等基础设施独立部署
- 业务模块在代码层面严格分包、分层、分领域隔离
- 跨模块联动优先通过事件总线，而不是数据库直连

## 4.3 未来拆分微服务的触发条件

满足以下条件中的任意两项及以上时，可以进入对应模块的服务化评估：

- 某模块有明显独立扩缩容需求
- 某模块吞吐量、延迟或资源占用已明显高于其他模块
- 某模块需要独立的发布节奏和故障隔离
- 某模块已形成清晰稳定的数据所有权和 API 边界
- 某模块已有相对独立的研发和运维责任边界

## 4.4 优先拆分候选顺序

后续如果要拆分，建议优先考虑：

1. `06-消息移动与生态`
2. `07-数据服务与集成`
3. `03-内容与知识` 中的搜索/推荐相关能力
4. `02-流程与表单` 中的流程引擎或任务中心（仅在规模足够大时）

不建议在一期就拆分：

- `01-组织与权限`
- `04-门户与工作台`

原因是这两个域在平台早期与全局上下文、权限判定和体验聚合耦合极强，过早拆分成本高、收益低。

## 5. 模块边界正式口径

## 5.1 `03-内容与知识` 与 `05-协同办公应用`

正式口径：

- `03` 拥有内容主模型、内容权限、内容检索、内容版本、订阅收藏等底座能力
- `05` 中涉及公告、网盘、文件协同等场景时，应优先复用 `03` 和 `00-附件中心` 的底座能力
- `05` 不再自建内容型对象的底层模型

## 5.2 `06-消息移动与生态` 与 `07-数据服务与集成`

正式口径：

- `06` 管用户触达、消息路由、移动端支持、SSO 与生态登录接入
- `07` 管开放 API、连接器、数据同步、服务治理、报表分析
- 回调、Webhook、开放平台授权、连接器参数等属于 `07`
- 渠道触达、消息模板、移动端聚合接口属于 `06`

## 5.3 `04-门户与工作台` 与 `07-数据服务与集成`

正式口径：

- `04` 负责面向用户首页与办公中心的聚合读模型
- `04` 不作为通用数据开放平台
- `07` 提供可复用、可授权、可治理的数据服务能力

## 5.4 `00-多租户` 与 `01-组织与权限`

正式口径：

- 租户：系统隔离、资源隔离、配置隔离、配额隔离
- 组织：业务主数据、管理结构、流程参与关系、授权主体
- 组织不等于租户，部门不等于租户层级

## 6. 权限与身份正式口径

## 6.1 权限模型

正式采用以下模型：

- 资源权限：RBAC
- 行级数据权限：ABAC/策略化授权
- 字段级数据权限：独立字段授权模型

授权主体统一支持：

- 组织
- 部门
- 岗位
- 人员
- 角色

## 6.2 身份上下文

正式采用以下规则：

- 人员必须且仅能有一个主岗
- 人员可以拥有多个兼岗
- 默认上下文为主岗
- 切换到兼岗后，菜单、待办、消息、数据范围、门户卡片、组织通讯录上下文都应同步刷新
- 所有权限判定均以当前身份上下文为准

## 6.3 权限判定顺序

正式采用以下顺序：

1. 身份上下文判定
2. 资源权限判定
3. 行级数据权限判定
4. 字段级数据权限判定

## 6.4 权限冲突规则

正式采用以下冲突规则：

- 显式拒绝优先于显式允许
- 更具体的授权主体优先于更宽泛主体
- 字段级限制优先于展示层默认能力
- 所有权限决策必须保留审计依据

## 10. 结论

本次收敛后，HJO2OA 的方向已经明确：

- 不直接进入全面微服务
- 先做模块化单体底座
- 先做统一组织权限、流程表单、门户聚合、消息触达闭环
- 先补文档契约，再做工程落地
- 通过边界收敛避免后续重复建设和模型冲突
