# ADR - 关键基础选型

## 1. 文档目的

本文档记录 HJO2OA 平台关键基础技术选型的架构决策记录（ADR），包括决策背景、候选方案、决策结果和决策理由。

对应架构决策编号：D18（关键基础选型必须形成 ADR）。

## 2. ADR 格式

每个 ADR 包含以下部分：

| 部分 | 说明 |
|------|------|
| 状态 | Proposed / Accepted / Deprecated / Superseded |
| 背景 | 决策背景和约束条件 |
| 候选方案 | 可选方案及其优缺点 |
| 决策结果 | 最终选型 |
| 理由 | 选择该方案的核心理由 |
| 影响 | 选型带来的影响和约束 |

---

## ADR-001 流程引擎选型

**状态**：Accepted

### 背景

流程引擎是 `02-流程与表单` 模块的核心，需支持复杂审批路由、会签/或签、条件分支、子流程、超时处理和动态参与者计算。引擎需与 Spring Boot 3.x / Java 17 兼容。

> **关联 ADR**：本选型受 ADR-006（流程与表单参考设计策略）约束——引擎选型需支撑"参考 O2OA 能力模型、自主定义领域模型与接口契约"的策略方向，而非直接复用 O2OA 引擎实现。

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Flowable** | BPMN 2.0 完整支持、社区活跃、文档丰富、支持 CMMN 和 DMN、与 Spring Boot 集成成熟 | 框架较重、历史 API 复杂、商业版功能更多 |
| **Camunda** | BPMN 2.0 完整支持、性能优秀、外部任务模式好、Operate/Tasklist 管理工具 | 社区版功能受限、商业版价格高、学习曲线陡 |
| **自研流程引擎** | 完全可控、轻量、可按需扩展 | 开发成本极高、需自行实现路由/会签/超时/监控、风险大 |

### 决策结果

**Flowable**（社区版）

### 理由

1. BPMN 2.0 标准支持完整，覆盖审批路由、会签/或签、条件分支、子流程等全部需求。
2. 与 Spring Boot 3.x 兼容，集成成本低。
3. 社区活跃，文档和示例丰富，降低团队学习成本。
4. 流程定义版本管理、流程实例挂起/恢复、任务转办/委派等能力开箱即用。
5. 自研引擎风险和成本远高于使用成熟框架。

### 影响

- 流程定义使用 BPMN 2.0 XML 格式，需设计器支持。
- 需封装 Flowable API 为平台统一接口，不直接暴露 Flowable 原生 API。
- 流程定义的参与者计算需自定义实现，与 `01-组织与权限` 的身份上下文集成。

## ADR-002 ORM 框架选型

**状态**：Accepted

### 背景

ORM 框架是后端数据访问层的基础，需支持复杂查询、多租户拦截、乐观锁、审计字段自动填充和分页。需与 Spring Boot 3.x / Java 17 兼容。

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **MyBatis Plus** | 国内生态成熟、SQL 可控性强、复杂查询灵活、学习成本低、多租户插件支持 | 无级联加载、实体映射需手写、代码量较大 |
| **Spring Data JPA** | 实体映射自动化、级联加载、审计字段自动填充、多租户过滤支持 | 复杂查询 N+1 问题、SQL 不可控、性能调优难、学习曲线陡 |

### 决策结果

**MyBatis Plus**

### 理由

1. 国内团队对 MyBatis 生态熟悉，学习成本最低。
2. SQL 完全可控，便于性能调优和复杂查询优化。
3. 多租户插件 `TenantLineInnerInterceptor` 开箱即用，与平台租户隔离需求匹配。
4. 乐观锁插件 `OptimisticLockerInnerInterceptor` 与接口契约的版本冲突策略匹配。
5. 分页插件 `PaginationInnerInterceptor` 与统一分页规范匹配。
6. JPA 的 N+1 问题和 SQL 不可控在 OA 场景下（大量列表查询和复杂筛选）风险较高。

### 影响

- 实体映射和级联查询需手写 Mapper。
- 审计字段（createdAt / updatedAt）需通过 MetaObjectHandler 自动填充。
- 领域模型中的值对象（JSON 字段）需自定义 TypeHandler 处理。

## ADR-003 消息队列选型

**状态**：Accepted

### 背景

消息队列是事件总线的底层基础设施，需支持可靠投递、消息持久化、消费确认、死信队列、延迟消息和分区顺序消费。

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **RabbitMQ** | 路由灵活、死信队列原生支持、延迟消息插件、管理界面友好、Spring AMQP 集成成熟 | 吞吐量低于 Kafka、消息堆积能力有限、集群运维复杂 |
| **Apache Kafka** | 超高吞吐、分区顺序保证、持久化强、生态丰富 | 运维复杂、无原生死信队列、延迟消息需自行实现、不适合低延迟场景 |
| **RocketMQ** | 事务消息原生支持、延迟消息、顺序消息、死信队列、国产生态 | 社区活跃度低于 Kafka/RabbitMQ、Spring 集成不如前两者成熟 |

### 决策结果

**RabbitMQ**

### 理由

1. OA 场景消息量级（万级/秒）远低于 Kafka 适用的百万级吞吐场景，RabbitMQ 性能足够。
2. 死信队列和延迟消息原生支持，与事件契约的补偿机制和超时催办需求完全匹配。
3. 路由灵活（Exchange/Binding 模式），与事件目录的多消费者订阅模式匹配。
4. Spring AMQP 集成成熟，开发成本低。
5. 管理界面友好，便于运维和问题排查。

### 影响

- 事件总线需基于 Spring AMQP 封装，提供统一发布/订阅 API。
- 消息持久化需配置为持久化模式（deliveryMode=2）。
- 消费者需手动 ACK，配合重试和死信队列策略。
- 未来如需升级到更高吞吐场景，可考虑迁移到 Kafka 或 RocketMQ。

## ADR-004 搜索引擎选型

**状态**：Accepted

### 背景

搜索引擎是 `03-内容与知识` 统一检索和 `04-门户与工作台` 聚合查询的基础，需支持全文检索、中文分词、聚合统计、实时索引和结构化过滤。

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Elasticsearch** | 全文检索能力强、中文分词插件丰富、聚合统计灵活、Spring Data ES 集成成熟 | 资源消耗大、集群运维复杂、与数据库双写一致性需处理 |
| **OpenSearch** | Elasticsearch 开源分支、功能等价、无许可证风险 | 社区生态略小于 ES |
| **PostgreSQL 全文检索** | 无额外中间件、运维简单、事务一致性天然保证 | 中文分词能力弱、性能低于专用搜索引擎、聚合统计能力有限 |

### 决策结果

**一期 PostgreSQL 全文检索，二期按需引入 Elasticsearch**

### 理由

1. 一期实施范围（`00/01/02/04/06`）对全文检索需求有限，PostgreSQL 的 `tsvector` + `pg_jieba` 分词可满足基础检索。
2. 避免一期引入 ES 带来的运维复杂度和双写一致性问题。
3. 二期引入 `03-内容与知识` 后，内容量和检索复杂度上升，再引入 Elasticsearch。
4. 搜索接口需抽象为统一层，底层实现可替换（PG -> ES 迁移对业务透明）。

### 影响

- 搜索接口必须抽象为统一 Service 层，不直接暴露底层实现。
- 一期使用 PostgreSQL 全文检索时，需安装 `pg_jieba` 或 `zhparser` 中文分词扩展。
- 二期引入 ES 时，需设计数据同步机制（基于事件总线的异步索引更新）。

## ADR-005 多租户隔离策略

**状态**：Accepted

### 背景

多租户是 SaaS 化交付的基础，需在数据隔离、资源隔离、配置隔离和配额隔离之间平衡成本、安全性和运维复杂度。

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **共享数据库 + 租户字段** | 成本最低、运维最简单、扩展性好 | 隔离性最弱、SQL 注入风险需额外防护、数据量大时性能下降 |
| **共享数据库 + 独立 Schema** | 隔离性中等、成本较低、可按租户备份恢复 | Schema 迁移复杂、连接池管理复杂、租户数量受限 |
| **独立数据库** | 隔离性最强、安全性最高、可按租户独立运维 | 成本最高、运维最复杂、跨租户查询困难 |

### 决策结果

**共享数据库 + 租户字段**（一期默认），**独立数据库**（政企大客户按需升级）

### 理由

1. 一期交付场景以中小客户为主，共享数据库 + 租户字段成本最低、运维最简单。
2. MyBatis Plus 的 `TenantLineInnerInterceptor` 可自动注入租户条件，开发成本低。
3. 政企大客户对数据隔离要求高，可按需升级为独立数据库模式。
4. 数据访问层需抽象为统一接口，底层隔离策略可按租户配置切换。

### 影响

- 所有业务表必须包含 `tenantId` 字段。
- 所有查询必须自动注入租户条件（通过 MyBatis Plus 拦截器）。
- 管理端跨租户查询需显式绕过租户拦截器（需超级管理员权限）。
- 独立数据库模式需设计数据源路由机制（如 `AbstractRoutingDataSource`）。
- 租户字段不得参与组织层级、岗位层级和授权主体计算（D06）。

## ADR-006 流程与表单参考设计策略

**状态**：Accepted

### 背景

HJO2OA 目标在业务能力上对齐 O2OA 核心平台，尤其是流程、表单、待办等协同底座；但整体架构已确定为模块化单体 + 独立基础设施服务，且平台需要遵守自主领域边界、统一身份/权限/事件契约，不能直接沿用 O2OA 的产品结构与源码组织方式。同时 O2OA 使用 AGPL-3.0 开源协议，直接复用源码将增加许可证与后续治理约束。

> **关联 ADR**：本策略约束 ADR-001（流程引擎选型）——无论选择哪种引擎，其使用方式必须符合"自主领域模型 + 统一契约"原则，而非直接映射 O2OA 引擎的产品结构。

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接二开 O2OA** | 功能现成、上线快、流程/表单/待办闭环完整 | 受 AGPL 约束、产品结构耦合重、与现有模块边界和统一契约不一致、历史技术负担 |
| **参考 O2OA 设计，自主实现** | 可借鉴成熟能力模型，同时保持领域边界、技术栈与接口契约统一 | 需要自行实现设计器/运行时/集成层，前期设计成本更高 |
| **完全不参考现有产品，自主探索** | 完全自主、无历史包袱 | 容易重复踩坑、流程与表单闭环设计风险高、收敛速度慢 |

### 决策结果

**参考 O2OA 的能力模型与交互闭环，自主定义领域模型、接口契约与实现结构**

### 理由

1. 能复用 O2OA 在流程定义/实例、表单元数据、统一待办、审计留痕等成熟设计经验。
2. 避免直接继承其产品化耦合和历史技术负担，保持与 HJO2OA 模块化单体架构一致。
3. 流程引擎、身份上下文、数据权限、消息提醒等关键边界需以 HJO2OA 的统一契约为准。
4. 直接复用源码会带来 AGPL 合规与后续定制治理成本。
5. 一期更需要能力闭环和边界清晰，而非完整复制低代码产品。

### 影响

- `02-流程与表单` 采用"设计时/运行时/读模型"分层，不直接映射 O2OA 的源码结构。
- 业务主数据归业务域所有，流程域仅维护流程状态、任务、动作和表单快照/引用。
- 身份解析、字段/行级权限判断、消息触达、门户聚合分别复用 `01`、`06`、`04` 的统一能力。
- 一期暂缓重脚本低代码、复杂门户深度联动、泛化数据中心等高复杂度特性。
