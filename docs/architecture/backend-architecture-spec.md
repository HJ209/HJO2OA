# 后端系统架构与规范

## 1. 文档目标

本文档用于定义 HJO2OA 后端系统的总体架构、技术边界、模块拆分原则、数据与接口规范、安全与运维规范。目标是在业务能力上对齐 O2OA 的核心平台能力，同时用更现代、可治理、可持续迭代的方式实现后端底座。

## 2. 后端建设原则

- 业务能力对齐 O2OA 核心平台，但不复制其历史技术负担。
- 先模块化单体，后按压力与边界拆分微服务。
- 以领域模型为中心组织代码，而不是以控制器或表名堆叠。
- 共享能力平台化，避免每个业务模块重复实现权限、消息、附件、审计等功能。
- 所有接口、权限、日志、错误码、审计、字典、租户隔离规则必须统一。
- 优先保证流程、内容、组织、门户等核心能力的稳定性与一致性。

## 3. 目标技术架构

### 3.1 架构形态

建议首期采用“模块化单体 + 独立基础设施服务”的方案。

- 业务服务以一个主应用承载，内部按领域模块严格隔离。
- 网关、对象存储、缓存、消息队列、搜索、文件预览等能力可作为外部基础设施独立部署。
- 只有在模块同时具备清晰数据边界、稳定 API 边界，并出现独立扩缩容、独立发布节奏或故障隔离诉求时，才进入服务化评估。
- 优先服务化候选可考虑消息移动与生态、数据服务与集成，以及内容搜索/推荐等高波动能力。

这样可以兼顾交付速度、复杂度控制和后续演进空间，适合从 0 到 1 复刻 O2OA 类平台。

### 3.2 确定技术栈

> 以下选型已通过 ADR 正式收敛，详见 `docs/decisions/adr-key-technology-selection.md`。

- Java 17（当前工程骨架基线，后续如需升级应通过 ADR 单独收敛）
- Spring Boot 3.x
- Spring Security
- MyBatis Plus（ADR-002）
- PostgreSQL 作为主数据库（ADR-004 一期全文检索亦基于 PG）
- Redis 作为缓存、会话、幂等、限流与临时状态存储
- RabbitMQ 作为异步消息总线（ADR-003）
- MinIO 或兼容 S3 的对象存储作为附件中心
- 一期 PostgreSQL 全文检索，二期按需引入 Elasticsearch（ADR-004）
- Flowable 社区版作为流程引擎（ADR-001，使用方式受 ADR-006 约束）
- Quartz 或 XXL-Job 作为任务调度能力
- Docker + CI/CD 作为标准交付方式

## 4. 核心后端分层

### 4.1 接入层

- API 网关
- Web 管理端接口
- 门户端接口
- 移动端接口
- 第三方开放接口
- 回调与 webhook 接口

职责是鉴权、限流、路由、灰度、请求日志和统一错误转换，不承载复杂业务。

### 4.2 应用层

- 负责用例编排、事务边界、权限校验、参数校验
- 不直接承载复杂领域规则
- 向前提供稳定的应用服务接口，向后调用领域服务与基础设施能力

### 4.3 领域层

- 承载组织、权限、流程、内容、门户、消息、文件、协同应用等核心领域规则
- 定义聚合、实体、值对象、领域服务和领域事件
- 维护业务不变量与跨对象的一致性

### 4.4 基础设施层

- 数据库持久化
- 搜索引擎适配
- 对象存储适配
- 消息队列适配
- 第三方登录与消息渠道适配
- 文件预览与格式转换能力适配
- 审计日志、分布式追踪、监控告警等横切能力

- `04-门户与工作台` 的聚合接口仅服务门户首页和办公中心场景，属于用户侧聚合读模型，不承担通用数据开放职责。
- `06-消息移动与生态` 负责用户触达、消息渠道、移动入口和生态登录接入；`07-数据服务与集成` 负责开放 API、连接器、数据同步、报表与服务治理。
- `00-多租户` 与 `01-组织与权限` 必须严格分离：租户是系统隔离边界，组织是业务管理结构，二者不得混用。

## 6. 事件驱动架构规范

### 6.1 事件总线

- 所有跨模块联动必须通过事件总线完成，禁止模块间直接调用
- 事件类型命名规范：`{模块前缀}.{事件主题}.{动作}`，如 `process.task.completed`、`todo.item.created`、`msg.notification.sent`
- 事件信封与载荷字段以《统一事件契约》为准，必须至少包含 `eventId`、`eventType`、`eventVersion`、`source`、`timestamp`、`correlationId`、`operatorAccountId`、`operatorPersonId`、`tenantId`、`traceId`、`payload`
- 事件投递保证至少一次，消费端必须幂等

### 6.2 事件生产者

- 各子模块在业务操作完成后发布领域事件，不关心谁消费
- 每个子模块的 `events/README.md` 声明其发布的领域事件
- 纯消费/查询类子模块不发布事件

### 6.3 事件消费者

- 消息中心、工作台、审计、数据同步等模块订阅感兴趣的事件
- 支持精确匹配和通配符匹配（如 `process.*`、`app.meeting.*`）
- 消费失败进入重试，超过重试次数进入死信队列

## 7. 国际化与时区规范

### 7.1 国际化（i18n）

- 前端文案、后端消息、错误提示统一走 i18n 资源包
- 资源包按模块/子模块分组，支持热更新
- 后端按 Accept-Language 返回对应语言
- 日期、数字、货币按 locale 格式化

### 7.2 时区管理

- 所有时间字段以 UTC 存储，查询时按用户时区转换
- 定时任务按指定时区触发，非服务器时区
- 日历/日程按用户时区展示

### 7.3 多语言数据（data-i18n）

- 需要多语言的业务字段（栏目名、标签名、公告标题等）按 `{entityId, field, locale}` 存储翻译值
- 查询时按用户 locale 返回对应语言值，无翻译时回退默认语言
- 翻译工作台管理待翻译条目和翻译状态

### 7.4 错误码国际化

- 所有错误码必须关联多语言消息
- 后端按 Accept-Language 返回对应语言的错误描述
- 前端统一拦截错误码并展示本地化提示

## 8. 模块拆分原则

- 一个模块只能拥有自己的业务主数据与规则，不直接写其他模块的数据表。
- 跨模块访问优先通过应用服务、领域服务或事件通知完成。
- 涉及统一组织、权限、消息、附件、审计时，必须调用平台级共享能力。
- 需要独立对外输出能力的模块，优先设计清晰 API，而不是直接暴露内部数据结构。
- 流程、内容、门户、数据服务属于平台基座，业务应用只能扩展，不得绕过。

## 9. 数据库设计规范

### 9.1 通用要求

- 所有业务表必须包含主键、创建人、创建时间、更新人、更新时间、删除标记、租户标识。
- 所有重要业务记录必须支持审计留痕和状态变化跟踪。
- 不允许以物理删除作为默认删除方式，优先逻辑删除与归档策略。
- 大字段、附件、全文内容与业务主表解耦。
- 查询条件高频字段必须建立索引。

### 9.2 命名规范

- 表名使用统一前缀与业务域标识。
- 字段名保持英文小写加下划线风格。
- 枚举值统一沉淀到字典配置或常量中心，不允许各模块自行散落定义。
- 外键关系尽量在应用层维护，减少强耦合迁移成本。

### 9.3 数据一致性策略

- 单模块内部优先本地事务。
- 跨模块场景优先事件驱动与最终一致性。
- 流程流转、审批记录、消息通知、索引刷新之间要有补偿机制。
- 所有异步任务都要有重试、幂等、死信和人工补偿入口。

## 10. 接口设计规范

### 10.1 API 分层

- 管理端 API
- 门户端 API
- 移动端 API
- 开放平台 API
- 内部服务 API

### 10.2 API 统一规则

- URL 语义清晰，按资源与动作组织
- 返回结构统一
- 错误码统一
- 分页、排序、过滤条件统一
- 所有写接口必须具备权限校验和操作审计
- 所有导入导出接口必须考虑数据量控制、异步任务和操作留痕

### 10.3 版本管理

- 外部 API 必须具备版本号策略
- 废弃接口要有过渡期和兼容说明
- 内部接口变更要建立契约检查机制

## 11. 权限与安全规范

### 11.1 权限模型

建议采用“组织 + 部门 + 岗位 + 人员 + 角色 + 资源权限 + 数据权限”的复合模型。

- 组织支持机构树，机构可以归属上级机构。
- 部门应归属机构，并支持部门层级。
- 岗位应同时具备机构归属和部门归属。
- 人员应同时具备机构归属和部门归属。
- 人员岗位需区分主岗和兼岗，主岗只能有一个且必选，兼岗可多个。
- 角色应优先与岗位关联，由人员通过岗位继承角色，直接对人员授予角色仅作为例外机制。
- 租户是系统隔离边界，不参与组织层级、岗位层级和授权主体计算；组织、部门、岗位、人员、角色属于业务授权结构，不得与租户层级混用。

在权限模式上，建议采用分层混合方案：

- 资源权限采用经典 RBAC，负责菜单、按钮、接口、页面与资源动作权限。
- 行级数据权限采用 ABAC 或策略化授权模型，授权主体支持组织、部门、岗位、人员、角色。
- 字段级数据权限采用独立字段授权模型，授权维度应至少包含业务对象、使用场景、字段、授权主体、权限动作。
- 对共享文档、网盘空间、知识空间、项目协作等强关系型场景，可按需补充 ReBAC 或 Fine-Grained Authorization 方案，但不建议在第一阶段作为全局基础模型。

推荐的权限判定顺序为：

- 先判定身份上下文，默认主岗，显式切换后使用兼岗上下文。
- 后台岗位、组织、账号状态变化导致当前上下文失效时，必须撤销旧上下文并发布 `org.identity-context.invalidated`。
- 再判定资源权限，判断用户是否有权进入模块、页面、接口或执行动作。
- 再判定行级数据权限，裁剪可访问记录范围。
- 最后判定字段级数据权限，裁剪可查看、可编辑、可导出的字段范围。
- 身份上下文显式切换后，菜单、待办、消息、数据范围、门户卡片和组织通讯录上下文都必须同步刷新。
- 后台发布 `org.identity-context.invalidated` 后，所有依赖身份上下文的读模型都必须沿用同一刷新链路完成重算；若不存在可恢复的有效上下文，则必须要求客户端强制重新登录。

推荐的策略冲突原则为：

- 显式拒绝优先于显式允许。
- 更具体的授权主体优先于更宽泛的授权主体。
- 字段级限制优先于页面展示层的默认能力。
- 所有权限决策都必须保留可审计依据。

### 11.2 安全要求

- 统一认证中心
- 支持账号密码、LDAP、AD、OAuth2、企业微信、钉钉等接入方式
- 支持多因子认证与高风险操作二次确认
- 接口级权限与数据级权限分离
- 敏感字段加密存储或脱敏返回
- 上传文件必须经过类型校验、病毒扫描和访问控制
- 审批、公文、合同等高敏感模块必须支持更严密审计

### 11.3 审计要求

- 登录、登出、鉴权失败、越权访问必须记录
- 核心业务变更必须记录操作前后摘要
- 文件下载、导出、打印、分享等行为必须可追溯
- 管理后台所有配置修改必须具备审计记录

## 12. 流程与表单引擎规范

- 流程定义、表单定义、流程实例、任务实例、审批记录、抄送记录、操作轨迹要分层建模。
- 表单字段元数据必须平台化，不允许业务模块自行拼装无统一规范的表单协议。
- 审批动作、路由规则、条件判断、会签、加签、退回、撤回、转办等能力应抽象为标准动作模型。
- 流程与业务单据要做到松耦合映射，避免流程引擎绑死具体业务表结构。
- 待办、已办、抄送、草稿、归档必须形成统一任务中心。

## 13. 内容管理规范

- 栏目、分类、内容、标签、权限、版本、发布状态、审核记录需分离建模。
- 内容正文、附件、封面、标签、扩展属性要支持灵活配置。
- 内容发布链路需支持草稿、审核、发布、下线、归档。
- 知识库与信息发布共用内容底座，但权限策略和呈现方式可分离。
- 全文检索、推荐、热点、收藏、订阅作为增强能力纳入统一模型。

## 14. 文件与附件中心规范

- 所有模块统一接入附件中心，不得私建附件逻辑。
- 文件上传、下载、预览、分享、转存、归档、版本管理能力统一。
- 附件与业务实体通过中间关系表解耦。
- 支持权限继承、外链时效控制、水印和操作留痕。
- 支持 Office、PDF、图片、音视频等常见格式的在线预览策略。

## 15. 消息与通知规范

- 统一站内消息中心
- 支持待办提醒、流程通知、内容通知、系统通知、预警告警
- 支持邮件、短信、企业微信、钉钉、APP Push 等多渠道分发
- 消息模板平台化管理
- 发送记录、送达状态、失败重试、用户已读状态统一跟踪

## 16. 搜索与报表规范

- 搜索能力统一接入搜索中心
- 支持组织、人员、流程单据、知识内容、附件、公文、合同等跨模块搜索
- 高级搜索应支持权限过滤与条件筛选
- 报表能力优先读模型化，不直接压业务主链路数据库
- 统计口径必须可定义、可审查、可追溯

## 17. 可观测性与运维规范

- 标准化日志采集、链路追踪、指标监控、健康检查
- 核心链路必须有告警阈值
- 任务失败、消息积压、流程卡点、索引异常、对象存储异常必须有告警
- 所有服务都需要环境配置分层管理
- 容器镜像、数据库变更、配置变更、发布记录必须可追溯

## 18. 测试与质量规范

- 核心领域服务必须具备单元测试
- 关键审批、权限、流程、导入导出、消息发送必须有集成测试
- API 需具备契约测试与回归测试
- 重大业务场景必须建立验收清单
- 高风险模块发布前必须做权限回归和性能验证

## 19. 演进路线建议

### 19.1 第一阶段

- 聚焦 `00`、`01`、`02`、`04`、`06`，完成统一组织权限、流程表单、门户工作台和消息移动的最小平台闭环

### 19.2 第二阶段

- 完成 `03-内容与知识`，接入统一内容底座、统一检索和内容型发布能力

### 19.3 第三阶段

- 完成 `05-协同办公应用` 中的公文、会议、日程、任务、考勤、合同与资产等高频业务应用

### 19.4 第四阶段

- 完成 `07-数据服务与集成`，建设开放接口、连接器、数据同步、报表分析与服务治理能力，并评估候选模块的服务拆分时机

### 19.5 第五阶段

- 完成智能知识问答、流程智能推荐、内容智能摘要等 AI 增强能力

## 20. 结论

后端设计应在能力边界上向 O2OA 对齐，在工程实现上向现代企业级平台靠拢。核心不是简单做一个 OA 系统，而是建设一个可持续承载多个协同场景、多个业务应用和多个交付项目的平台型后端底座。
