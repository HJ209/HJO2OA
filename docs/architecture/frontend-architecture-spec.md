# 前端系统架构与规范

## 1. 文档目标

本文档用于定义 HJO2OA 前端系统的建设原则、应用架构、模块边界、交互规范、工程规范和发布规范。目标是在功能范围上支撑 O2OA 类协同办公平台的核心场景，同时以现代化前端体系提供更好的可维护性、性能、体验和二次开发效率。

## 2. 前端建设原则

- 对齐 O2OA 核心业务能力，但前端实现坚持现代化重构。
- 平台能力优先于页面堆叠，优先建设统一工作台、统一表单、统一消息、统一搜索、统一附件能力。
- 布局设计参考 O2OA 的门户与办公中心模式，门户首页强调顶部品牌区、中部多分区聚合区和底部信息区，办公中心强调左侧导航或分页切换、右侧工作列表与处理区。
- 多端统一设计语言、统一数据契约、统一权限反馈和统一异常处理。
- 页面、组件、状态、路由、接口、权限必须有清晰边界。
- 管理端、门户端、移动端共享模型但不强行共用所有实现。
- 复杂业务场景尽量组件化、配置化、元数据化。

## 3. 推荐前端技术栈

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Axios 或同类请求封装方案
- Element Plus、Naive UI 或企业内部设计系统
- ECharts 用于统计与图表展示
- VueUse 等基础工具库
- UnoCSS 或 TailwindCSS，用于提升样式组织效率
- PWA 能力用于移动办公增强

## 4. 前端应用形态

### 4.1 管理端

面向平台管理员、流程管理员、内容管理员、运维管理员、应用管理员。负责组织权限、流程设计、门户配置、内容栏目管理、字典配置、系统配置、审计查看、集成配置等场景。

### 4.2 门户端

面向普通员工和业务人员。负责工作台、待办、已办、消息、公告、知识库、文件、日程、会议、公文、任务等日常办公场景。

### 4.3 移动端

面向移动审批、消息触达、轻量操作和移动办公。移动端优先覆盖高频链路，如登录、待办处理、流程发起、消息查看、会议签到、日程提醒、内容阅读、文件预览等能力。

### 4.4 低代码设计端

面向流程设计、表单设计、门户设计等平台型能力，作为独立子应用建设，避免与普通业务页面耦合。

## 5. 前端总体架构

### 5.1 应用组织方式

建议采用 Monorepo 或至少统一工程规范的多应用仓库结构，按应用角色拆分为：

- 管理后台
- 门户工作台
- 移动端 H5
- 设计器子应用
- 共享组件库
- 共享工具库
- 类型定义与接口契约库

### 5.2 分层结构

- 应用壳层：布局、导航、主题、国际化、权限入口
- 页面层：具体业务页面与用例编排
- 领域组件层：组织树、审批流、附件面板、表单渲染器、消息列表等业务复用组件
- 基础组件层：按钮、表单控件、弹窗、表格、卡片、上传器、图表容器等
- 数据访问层：请求封装、错误处理、缓存与重试策略
- 状态层：用户态、组织态、权限态、消息态、搜索态、工作台态等

### 5.3 参考 O2OA 的页面布局模式

- 门户首页采用“顶部品牌或导航区 + 中部多区块内容区 + 底部版权或说明区”的三段式结构，中部内容区可继续拆为上中下多行布局。
- 办公中心采用“左侧功能导航或分页切换 + 右侧主工作区”的双栏结构，左侧承载待办、已办、待阅、已阅、草稿等工作入口，右侧承载列表、详情和处理动作。
- 业务发起页采用“顶部筛选或说明区 + 中部列表或卡片区”的模式，流程发起类页面可按组织、分类、科室或业务域分组展示可发起事项。
- 门户设计器采用“左侧组件区 + 顶部工具栏 + 中部画布区 + 右侧属性区”的设计器模式，贴近 O2OA 门户开发界面。
- 门户内容装配以部件化为主，待办、公告、制度、图片新闻、统计图卡都应作为可复用部件配置，而不是写死在首页。

## 7. 国际化与时区规范

### 7.1 国际化（i18n）

- 所有用户可见文案必须走 i18n 资源包，禁止硬编码中文
- 语言切换后全局刷新，包括菜单、提示、表单校验消息
- 日期、数字、货币使用 locale 格式化组件
- 资源包按模块分组，支持按需加载和热更新

### 7.2 时区管理

- 所有时间展示必须按用户时区转换，禁止直接展示 UTC
- 日历/日程组件必须适配时区
- 跨时区协作场景标注时区信息

### 7.3 多语言数据

- 业务数据多语言字段使用语言 Tab 切换编辑
- 列表页自动展示当前语言数据
- 翻译工作台供翻译人员使用

### 7.4 错误码国际化

- 前端统一错误拦截器，按错误码展示本地化提示
- 错误码文档页支持多语言切换

## 8. 路由与导航规范

- 路由按业务域组织，不按页面类型随意堆叠。
- 一级导航以平台能力和业务中心划分，优先支持顶部主导航与左侧业务导航的组合模式。
- 二级导航根据用户身份和权限动态呈现，办公中心类页面优先呈现为左侧工作分类导航。
- 页面面包屑、返回逻辑和标签页逻辑必须统一。
- 所有页面都要支持从消息、待办、搜索结果、收藏、快捷入口等多入口进入。
- 门户首页、办公中心、业务二级页应共享同一应用壳能力，但允许采用不同内容布局模板。

## 9. 状态管理规范

- 用户身份、当前组织、当前租户、菜单权限、按钮权限、字典缓存、主题配置等为全局状态。
- 页面筛选条件、表格状态、弹窗开关等优先就地管理，避免滥用全局状态。
- 待办、消息、搜索、通知等跨页面状态统一沉淀为平台级 store。
- 所有缓存策略必须有失效规则与刷新机制。

## 10. 页面设计规范

### 10.1 页面骨架规范

- 门户类页面优先采用顶部导航区、内容区、底部信息区三段式结构。
- 应用类页面优先采用顶部操作区、左侧导航区、右侧主工作区布局。
- 顶部区域应统一承载品牌、身份切换、全局搜索、消息入口和快捷操作。
- 页面宽度和栅格应同时支持门户多栏内容编排与工作区单主视图切换。

### 10.2 门户首页规范

- 首页应以信息聚合和快速进入为目标，布局上参考 O2OA 常见门户首页模式。
- 首屏优先展示 banner、待办待阅、通知公告、规章制度、图片新闻和统计区域。
- 中部内容区建议按上中下分段，每段再按左右栏或单栏组合。
- 各区块以卡片或部件形式装配，支持拖拽、排序和按角色模板替换。
- 底部可保留版权信息、系统说明或友情链接区域。

### 10.3 办公中心与工作区规范

- 办公中心优先采用左侧导航、右侧工作区模式。
- 左侧导航用于待办、已办、待阅、已阅、草稿、收藏等工作入口切换。
- 右侧工作区用于列表、详情、审批处理和业务操作。
- 列表点击后应支持在当前主工作区展开详情或跳转处理页，避免层层弹窗。
- 顶部应保留新建流程、新建业务、批量处理等主操作入口。

### 10.4 列表页规范

- 顶部统一查询区
- 中部统一表格或卡片区
- 右上统一主操作入口
- 批量操作、导出、导入、筛选、排序能力保持统一交互
- 高风险操作必须二次确认

### 10.5 详情页规范

- 业务主信息区与操作区清晰分离
- 审批记录、操作日志、附件、关联对象采用统一侧栏或分区样式
- 支持上下文返回，避免用户迷失页面层级

### 10.6 表单页规范

- 创建、编辑、查看、审批四种状态要有明确呈现差异
- 表单字段权限、只读规则、必填规则、联动规则由统一协议驱动
- 草稿保存、提交校验、审批意见、附件上传能力统一

### 10.7 工作台规范

- 支持待办、常用应用、消息、日程、公告、统计图卡等模块化布局
- 支持个人定制与管理员预置模板
- 支持不同组织、岗位、角色的差异化工作台

### 10.8 设计器页面规范

- 门户设计器、表单设计器、流程设计器优先采用左侧组件区、顶部工具栏、中部画布区、右侧属性区的统一结构。
- 数据源、事件脚本、资源文件、页面属性等配置能力应与主画布保持弱耦合分区展示。
- 设计器需支持实时预览、草稿保存、模板复用和多端预览。

## 11. 组件设计规范

- 基础组件无业务耦合
- 领域组件聚焦平台共享能力
- 页面组件只承载当前页面用例，不反向污染共享层
- 所有通用组件必须定义明确输入输出协议
- 高复用组件优先支持配置化而非复制页面逻辑

重点建设的领域组件包括：

- 组织树选择器
- 人员与身份选择器
- 权限标签与权限提示组件
- 动态表单渲染器
- 流程轨迹组件
- 审批动作组件
- 附件上传与预览组件
- 内容阅读器
- 工作台卡片组件
- 统一消息项组件
- 门户区块容器组件
- 左导航工作区组件
- 设计器属性面板组件

## 12. 接口调用规范

- 所有接口通过统一请求层调用
- 统一处理 token 续期、权限异常、网络错误、业务错误
- 支持请求取消、重复请求去重、幂等防抖和失败重试策略
- 上传下载、长轮询、流式返回、批量导出需单独封装
- 前端不得自行拼装未定义字段或依赖后端隐式返回结构

## 13. 权限前端规范

- 菜单级权限控制负责“可见性”
- 按钮级权限控制负责“可操作性”
- 数据级权限由后端主导，前端负责合理展示和提示
- 越权时要明确提示，不可静默失败
- 身份切换、岗位切换、组织切换后必须刷新权限上下文，同时菜单、待办、消息、数据范围、门户卡片和组织通讯录上下文都必须同步刷新
- 收到 `org.identity-context.invalidated` 时，前端必须沿用同一刷新链路回收本地身份缓存并重新拉取菜单、待办、消息、数据范围、门户卡片和组织通讯录上下文；若无法恢复有效上下文，则必须进入强制重新登录

## 14. 表单与设计器规范

- 表单渲染器与表单设计器分离
- 字段协议、校验协议、布局协议、联动协议统一建模
- 流程表单、业务表单、只读详情表单尽可能共用渲染底座
- 设计器输出的是元数据，不是页面源码
- 表单中嵌入附件、子表、审批意见、人员选择、字典选择等能力统一实现

## 15. 内容与门户规范

- 内容阅读页支持正文、目录、附件、评论、推荐、收藏、订阅等增强能力
- 门户页面支持卡片、图文、轮播、导航、图表、待办、公告等部件化装配
- 门户页面应优先覆盖系统首页、列表页、二级页、部门门户和报表门户等 O2OA 常见页面类型
- 门户首页推荐采用 banner、信息聚合区、统计区、底部说明区的组合模式
- 门户设计需兼顾 PC 与移动端自适应策略
- 内容权限、门户权限、应用权限之间关系要清晰，不允许前端自行推断授权

## 16. 性能优化规范

- 路由级懒加载
- 大表格虚拟滚动或分页优化
- 大型门户页面按部件分片加载
- 高频字典与组织数据合理缓存
- 重型设计器按需拆包
- 图片、附件缩略图、图表资源要做懒加载与缓存策略

## 17. 可用性与体验规范

- 所有关键操作要有明确反馈
- 空状态、加载状态、错误状态、权限不足状态都必须统一设计
- 审批、上传、发布、导入、导出等长流程操作必须有进度与结果反馈
- 兼顾键盘操作、可读性、响应速度与移动触控体验
- 复杂业务页面优先保证信息密度和操作效率，不追求表面花哨

## 18. 工程规范

- 全量 TypeScript 化
- ESLint、Prettier、Stylelint 统一规范
- 组件、页面、store、api、types、composables 目录职责清晰
- 严格禁止跨层级随意引用
- 公共类型定义集中管理
- 环境变量、接口地址、功能开关统一管理

## 19. 测试与交付规范

- 关键组件具备单元测试
- 核心业务链路具备端到端测试
- 接口契约变更需同步检查前端影响面
- 视觉回归测试优先覆盖工作台、流程表单、内容详情、门户首页等复杂页面
- 发布采用环境隔离与灰度策略

## 20. 演进建议

### 20.1 第一阶段

- 聚焦 `00`、`01`、`02`、`04`、`06`，完成登录认证、门户壳、工作台、待办中心、消息中心、基础组织与权限、流程发起与审批、表单渲染器的最小平台闭环

### 20.2 第二阶段

- 完成 `03-内容与知识`，接入内容中心、附件中心、搜索中心

### 20.3 第三阶段

### 20.4 第四阶段

- 完成 `07-数据服务与集成`，建设门户设计器、表单设计器、数据服务管理、第三方集成管理，并评估候选模块的服务拆分时机

### 20.5 第五阶段

- AI 助手、知识问答、智能推荐、智能摘要等增强体验

## 21. 模块边界口径

- `03-内容与知识` 拥有内容主模型和统一检索模型，`05-协同办公应用` 中公告与文件协同场景必须优先复用 `03` 与 `00-附件中心` 的底座能力。
- `04-门户与工作台` 的聚合接口仅服务门户首页和办公中心场景，属于用户侧聚合读模型，不承担通用数据开放职责。
- `06-消息移动与生态` 负责用户触达、消息渠道、移动入口和生态登录接入；`07-数据服务与集成` 负责开放 API、连接器、数据同步、报表与服务治理。
- 租户是系统隔离边界，组织是业务管理结构，前端不得将租户选择与组织层级混用。

## 22. 结论

前端系统的核心目标不是简单堆页面，而是形成一套能够长期承载 OA、协同办公、低代码配置和行业业务扩展的统一前端平台。只有平台能力统一，前后端协同才不会在复刻 O2OA 的过程中再次陷入复杂度失控。
