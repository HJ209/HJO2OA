# 03-内容与知识 模块设计

## 1. 文档目的

本文档定义 `03-内容与知识` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围。作为模块功能文档、领域模型文档和开发任务文档的上层约束文档。

对应架构决策编号：D03（`03` 与 `05` 模块边界）、D12（事件契约）、D14（内容底座约束）、D20（一期实施范围）、ADR-006（参考设计策略）。

## 2. 设计策略

### 2.1 参考设计原则

本模块参考 O2OA 的信息发布、知识库、栏目管理和检索订阅能力闭环，但不沿用其产品结构和重耦合实现方式（ADR-006）。具体原则：

- **借鉴统一内容底座**：新闻、公告、制度、知识、专题等内容对象共享一套主模型、版本和发布能力。
- **不借鉴业务侧重复建模**：公告、共享文件、制度发布等场景不再在 `05` 或其他业务域内部复制内容底层模型（D03、D14）。
- **不借鉴搜索替代主数据**：全文索引、推荐和统计看板都必须以内容主模型为源，不得反向成为真相源。
- **优先统一发布与可见性规则**：先收敛栏目权限、发布范围、审核状态和检索索引规则，再扩展到复杂运营功能。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| 内容底座拥有内容主模型 | 稿件、版本、发布、订阅收藏、检索索引归 `03` |
| 附件底座仍归 `00` 所有 | `03` 只管理附件绑定和正文引用，不持有文件二进制 |
| 审核流程可接 `02`，但发布状态归 `03` | 工作流是审核机制，不是内容状态机 |
| 门户聚合和消息触达不归内容域 | `04` 负责聚合展示，`06` 负责通知发送 |
| `05` 场景复用 `03`，不自建内容底层模型 | 公告、知识展示、共享文件目录等必须复用内容底座 |
| 搜索与统计是投影能力 | 全文索引、热度排行和推荐结果不得直接替代内容主数据 |

### 2.3 O2OA 能力借鉴清单

| O2OA 能力 | HJO2OA 一期保留 | HJO2OA 一期简化 | HJO2OA 暂缓 |
|-----------|----------------|----------------|-------------|
| 栏目/分类/标签/专题管理 | ✅ | - | - |
| 稿件生命周期（草稿/送审/发布/归档） | ✅ | - | - |
| 正文、封面、附件与版本快照 | ✅ | - | - |
| 发布范围与可见性控制 | ✅ | - | - |
| 全文检索与分类筛选 | ✅ | 基础检索 | 复杂推荐 |
| 收藏与订阅 | ✅ | 基础订阅 | 复杂画像/订阅编排 |
| 阅读统计与热度排行 | ✅ | 基础统计 | 高级运营分析 |
| 多语言内容运营 | - | 基础多语言预留 | 深度多语言运营 |
| 知识图谱/复杂专题页 | - | - | ✅ |

## 3. 分层架构

### 3.1 语义组织层（Taxonomy Layer）

负责栏目、分类、标签、专题及可见规则的统一组织骨架。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `category-management/` | 栏目树、分类树、标签组、专题骨架、栏目权限规则 | `content_category`、`content_category_permission`、`content_tag_def`、`content_topic_def` |

**语义组织层规则**：

- 栏目、分类、标签、专题必须使用统一编码规则和统一可见性规则。
- 栏目权限是内容可见性的第一道边界，标签与专题不替代权限规则。
- 组织结构变更后，栏目可见范围缓存应可刷新，但不回写历史内容主数据。

### 3.2 内容主模型与生命周期层（Content Core Layer）

负责内容主模型、正文版本、审核、发布、下线和归档。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `content-lifecycle/` | 稿件创建、送审、发布、下线、归档、审核记录 | `content_article`、`content_publication`、`content_review_record` |
| `content-storage/` | 正文快照、封面引用、附件绑定、版本关系、内容引用 | `content_article_version`、`content_article_body`、`content_article_attachment`、`content_article_relation` |

**主模型与生命周期层规则**：

- 内容稿件与正文版本解耦，已发布版本不可原地编辑。
- 发布对象是具体版本，不是草稿状态或前台渲染快照。
- 正文、附件和引用关系均绑定到版本，保证历史可回溯。
- 归档只改变可见状态，不删除内容历史与审计轨迹。

### 3.3 发布权限与检索投影层（Publication & Search Layer）

负责发布范围、可见性校验和全文检索投影。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `content-permission/` | 发布范围、栏目权限、主体匹配、显式拒绝与允许裁决 | `content_publication_scope` |
| `content-search/` | 全文索引、分类筛选、相关推荐、收藏与订阅配置 | `content_search_index`、`content_subscription_profile`、`content_subscription_target`、`content_favorite_record` |

**发布权限与检索层规则**：

- 可见性判断至少同时经过栏目权限与发布范围两层校验。
- 只有 `PUBLISHED` 且当前有效的内容才进入检索索引。
- 检索索引是内容发布后的投影，不直接承载编辑能力。
- 收藏与订阅属于用户偏好，不能突破内容可见性边界。

### 3.4 运营统计层（Operation Insight Layer）

负责阅读、下载、收藏等行为的统计和运营分析。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `content-statistics/` | 阅读行为记录、热度排行、发布效果分析、订阅转化分析 | `content_read_record`、`content_engagement_snapshot` |

**运营统计层规则**：

- 行为明细采用追加式记录，统计结果允许异步聚合。
- 统计看板只能消费已发布内容，不对草稿和审核态内容做公开展示。
- 热度排行与运营图表是读模型，不回写影响内容主模型状态。

## 5. 跨模块集成

### 5.1 依赖关系

| 依赖方向 | 依赖内容 | 集成方式 |
|----------|----------|----------|
| `03 -> 00` | 附件中心 | 通过 `infra.attachment` 维护封面和正文附件引用 |
| `03 -> 00` | 审计、配置、字典、i18n、错误码 | 记录操作留痕、读取发布配置、复用基础字典与多语言能力 |
| `03 -> 01` | 身份上下文、组织/岗位/角色、数据权限 | 解析栏目权限和发布范围 |
| `03 -> 02` | 内容审核流程 | `WORKFLOW` 审核模式下消费流程结果 |
| `04 -> 03` | 公告/内容聚合查询 | `portal-home` 和 `aggregation-api` 消费已发布内容 |
| `05 -> 03` | 公告、知识、共享文件目录场景 | 业务场景复用内容底座，不再自建内容主模型 |
| `06 -> 03` | 发布通知、订阅触达 | 消费内容发布事件生成通知 |
| `07 -> 03` | 开放查询、报表分析、外部同步 | 通过数据服务和开放接口对外提供内容数据 |

### 5.2 事件契约

本模块发布的跨模块事件遵循 `docs/contracts/unified-event-contract.md`，模块前缀为 `content`。

| 事件类型 | 触发时机 | 消费模块 |
|----------|----------|----------|
| `content.category.created` | 栏目/分类创建 | 04/06 |
| `content.category.updated` | 栏目/分类更新 | 04/06 |
| `content.article.created` | 稿件创建 | 04/06 |
| `content.article.updated` | 稿件更新 | 04/06 |
| `content.article.published` | 内容正式发布 | 04/06/07 |
| `content.article.archived` | 内容归档 | 04/06/07 |

### 5.3 与流程审批的关系

| 场景 | 归属模块 | 说明 |
|------|----------|------|
| 内容状态机 | `03` | 草稿、送审、发布、下线、归档归内容域维护 |
| 审核流程编排 | `02` | 审核人、任务流转、审批动作由流程域承担 |
| 审核结果回写 | `03` 消费 `02` 事件 | 审核通过/驳回最终回写到内容发布记录 |

## 7. 发布、检索与通知策略

### 7.1 发布与下线策略

- 发布必须基于具体版本，审核通过后方可生效。
- 支持立即发布与定时发布，过期后自动下线或转归档。
- 已发布版本重新编辑时必须派生新草稿版本，不能直接覆写线上版本。

### 7.2 检索与索引策略

- 一期检索基于 SQL Server Full-Text Search 实现标题、摘要、正文与标签检索。
- 检索索引由内容发布事件驱动刷新，不依赖前台读取时实时构建。
- 下线和归档事件必须触发索引移除或降权处理。

### 7.3 订阅与通知策略

- `03` 负责计算订阅命中关系，`06` 负责消息模板、发送渠道和送达状态。
- 收藏与订阅变更优先写入主库，再异步更新统计和推荐投影。
- 一期不做复杂订阅编排，只支持实时或摘要两类策略。

## 8. 一期交付范围

### 8.1 一期平台口径

- `03` 不属于平台最小闭环必做模块（D20）。
- 当前设计用于保证后续按场景接入时边界清晰、模型稳定。

### 8.2 若项目提前接入，一期建议包含

- `category-management`：栏目、分类、标签、专题和栏目权限骨架
- `content-lifecycle`：稿件、送审、发布、下线、归档
- `content-storage`：正文快照、封面、附件绑定
- `content-permission`：发布范围和可见性校验
- `content-search`：基础全文检索、收藏和订阅

### 8.3 若项目提前接入，一期简化

- `content-statistics`：先支持阅读、下载、收藏的基础累计和热度排行
- 审核模式：先支持直接发布和单条工作流审核，不做复杂多级审核编排
- 搜索：先支持基础全文索引，不做复杂推荐与智能召回

### 8.4 一期暂缓

- 复杂专题推荐页和专题运营编排
- 高级推荐、知识图谱和语义关联
- 多语言全文检索与跨语种运营分析
- 深度内容运营看板和转化漏斗分析

## 9. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| `05` 或其他业务域重新建设公告/知识模型 | 强制所有内容型对象统一接入 `03`，场景域只保留业务装配能力 |
| 审核状态和发布状态耦合不清 | 审核流程归 `02`，发布状态机归 `03`，通过显式回写解耦 |
| 搜索索引与内容主数据不一致 | 统一由内容事件驱动索引刷新，消费端幂等并具备补偿入口 |
| 附件删除导致内容引用悬挂 | 附件绑定显式建模，并消费附件事件做一致性校验 |
| 统计与推荐过早复杂化 | 一期只保留基础统计和基础检索，复杂算法后置 |
