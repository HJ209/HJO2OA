# content-lifecycle 子模块总览

## 模块定位

`content-lifecycle` 位于内容主模型与生命周期层，负责稿件创建、送审、发布、下线、归档以及审核记录管理，是内容域状态机的唯一真相源。

## 核心职责

| 核心职责 | 说明 | 输出结果 |
|----------|------|----------|
| 稿件主模型管理 | 维护稿件、状态、所属栏目与当前生效版本引用。 | 形成统一内容主模型 |
| 审核与发布状态机 | 承接送审、审核、发布、下线和归档动作。 | 形成受控的生命周期轨迹 |
| 发布记录留痕 | 保存审核意见、发布时间、操作者和发布批次。 | 为检索、消息和审计提供事实依据 |

## 上下游边界

| 方向 | 模块/能力 | 边界说明 |
|------|-----------|----------|
| 上游 | `category-management`、`content-storage` | 引用栏目和版本快照，不管理其内部配置细节 |
| 上游 | `02-WorkflowForm` | 可复用流程做审核，但内容状态机仍归本模块所有 |
| 下游 | `content-permission`、`content-search`、`content-statistics` | 发布与下线事件驱动下游投影和统计更新 |
| 外部 | `04-Portal`、`06-Messaging` | 只暴露已发布内容，不负责首页编排与通知送达 |

## 核心能力

| 能力 | 说明 | 关键约束 |
|------|------|----------|
| 草稿创建与保存 | 支持建稿、修改元数据、选择所属栏目和版本草稿。 | 草稿可反复保存，发布版不可原地修改 |
| 审核与送审 | 支持人工审核和复用工作流的送审模式。 | 流程结果只影响审核结论，不替代内容状态机 |
| 发布/下线/归档 | 控制版本生效、失效和归档可见性。 | 发布对象是具体版本，归档不删除历史 |
| 版本回溯 | 查看历史发布批次和审核轨迹。 | 一期提供查看与比对，不提供任意时点恢复为新草稿 |

## 典型业务流

1. 编辑创建草稿并绑定栏目、分类与正文版本。
2. 根据审核模式提交审核或发起工作流。
3. 审核通过后选择具体版本发布，生成发布记录与事件。
4. 下线或归档时更新内容状态，并通知检索、统计与消息子模块刷新投影。

## 一期实现约束

- 已发布版本不可原地编辑，修改必须基于新草稿版本重新进入审核链路。
- 工作流是审核机制而非内容真相源，发布状态最终仍以内容域记录为准。
- 一期不做复杂编委会、多阶段串并审编排，保留基础审核记录和流程接入。
