# 06-消息移动与生态 模块设计

## 1. 文档目的

本文档定义 `06-消息移动与生态` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围，作为模块功能、领域模型和开发任务的上层约束文档。

对应架构决策编号：D04（`06` 与 `07` 模块边界）、D08（身份上下文）、D13（最终一致性）、D16（待办与消息分工）、D20（一期实施范围）。

## 2. 设计策略

### 2.1 参考设计原则

本模块遵循“事件驱动生成提醒、渠道投递与消息留痕分离、移动高频链路单独优化”的设计思路，具体原则如下：

- **先统一消息语义，再扩展渠道**：先把业务事件收敛为统一消息与提醒模型，再挂接站内信、邮件、短信、企业微信、钉钉、APP Push 等渠道。
- **先保证站内入箱，再追求外部触达增强**：一期所有消息至少进入站内消息中心，外部渠道作为增强触达能力。
- **先区分待办与消息，再做联动**：待办由 `02-todo-center` 持有事务处理真相，消息由 `06-message-center` 持有通知触达真相。
- **先优化移动高频场景，再避免复制 PC**：移动端聚焦消息、待办、审批、阅读、预览等高频链路，不复制一套独立门户底座。
- **生态接入以连接配置和状态治理为主**：`ecosystem` 负责企业微信、钉钉、SSO、LDAP 等接入配置、回调校验和状态监控，不替代 `07` 的开放接口和通用连接器治理。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| 消息不是待办 | 待办负责事务处理，消息负责触达提醒；两者联动但不混用状态 |
| 触达不是开放能力 | 本模块负责用户触达和移动入口，不负责开放 API、Webhook 和跨系统数据交换 |
| 站内消息与渠道投递分层 | 站内入箱状态与外部送达状态分别建模，避免状态污染 |
| 身份快照只引用 `01` 真相 | 接收人、任职、岗位、身份切换统一引用 `01` 的主数据与上下文 |
| 移动端不拥有业务主数据 | `mobile-support` 只承接设备、会话、高频 API 和弱网容错，不拥有流程、内容或业务主数据 |
| 生态接入不反向吞并同步职责 | LDAP/SSO/企微/钉钉接入归 `06`，但组织主数据同步落库和通用连接器治理不归 `06` |

### 2.3 一期能力收敛口径

| 能力 | 一期保留 | 一期简化 | 一期暂缓 |
|------|----------|----------|----------|
| 统一消息中心 | 是 | 聚焦站内消息、已读/归档/撤回、来源事件关联 | - |
| 多渠道发送 | 是 | 先支持基础路由、模板、失败重试和送达追踪 | - |
| 事件订阅与待办提醒 | 是 | 先聚焦 `todo.*`、`process.*` 等主链路事件 | - |
| 移动端接入 | 是 | 聚焦消息、待办、审批、阅读、预览等高频场景 | - |
| 生态接入 | 是 | 一期至少打通一种生态渠道或登录方式 | - |
| 模板设计与路由编排 | 是 | 基础变量替换与简单升级策略，不做复杂可视化编排 | - |
| 周期 Digest、复杂升级链路 | - | - | 是 |
| 全量生态矩阵与复杂离线补偿 | - | - | 是 |

## 3. 分层架构

### 3.1 事件接入与订阅层（Subscription Layer）

负责消费平台业务事件、计算目标接收人、生成提醒规则和订阅偏好。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `event-subscription/` | 事件接入、订阅规则、订阅偏好、静默规则、优先级升级 | `msg_subscription_rule`、`msg_subscription_preference` |

**事件接入与订阅层规则**：

- 只消费统一事件契约中已注册的业务事件，不依赖发布方私有内部模型。
- 待办相关提醒优先解析 `assignmentId`、`positionId`，保证消息与身份上下文对齐。
- 同一事件命中多条规则时必须经过 `dedupKey` 去重，避免重复轰炸用户。
- 订阅偏好只能覆盖非强制触达场景，不能完全静默安全告警等系统强制消息。

### 3.2 消息与投递层（Notification & Delivery Layer）

负责统一消息入箱、模板渲染、渠道路由、送达追踪、失败重试和补偿。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `message-center/` | 统一消息模型、已读/归档/撤回/过期状态管理、站内列表查询 | `msg_notification`、`msg_notification_action` |
| `channel-sender/` | 模板、渠道端点、路由策略、投递任务、送达追踪、重试补偿 | `msg_template`、`msg_channel_endpoint`、`msg_routing_policy`、`msg_delivery_task`、`msg_delivery_attempt` |

**消息与投递层规则**：

- 消息以“单接收人单消息”为粒度创建，不采用多人共享一条消息的设计。
- 站内入箱状态与外部渠道送达状态分离建模，统一归并为最终 `deliveryStatus`。
- 一期默认所有消息必须至少进入 `INBOX`，外部渠道作为路由策略决定的增强链路。
- 外部渠道失败不得回滚站内消息入箱，只影响投递状态、重试和补偿。

### 3.3 移动接入层（Mobile Access Layer）

负责移动端设备接入、会话续期、高频接口和弱网容错。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `mobile-support/` | 设备绑定、Push Token、移动会话、轻量 API、弱网容错 | `msg_device_binding`、`msg_mobile_session` |

**移动接入层规则**：

- 设备绑定只描述终端接入状态，不拥有身份上下文真值。
- 移动会话中的当前岗位/任职是快照，不替代 `01-identity-context` 的权威计算结果。
- 移动端应围绕待办、审批、阅读、预览等高频链路做轻量聚合，不复制桌面端的完整页面模型。
- 设备丢失、撤销或高风险冻结时必须使对应 Push Token 和会话快速失效。

### 3.4 生态接入与治理层（Ecosystem & Governance Layer）

负责第三方生态连接配置、回调校验、状态监控和接入治理。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `ecosystem/` | 企业微信、钉钉、SSO、LDAP、短信、邮件接入配置与监控 | `msg_ecosystem_integration`、`msg_callback_audit` |

**生态接入与治理层规则**：

- 生态接入配置只保存连接参数引用、回调校验规则和健康状态，不保存敏感明文。
- 消息实际发送仍通过 `channel-sender` 执行，`ecosystem` 更偏向接入治理和监控视角。
- 生态回调、连接测试和异常排查都必须进入统一审计和状态监控。

## 5. 跨模块集成

### 5.1 依赖关系

| 依赖方向 | 依赖内容 | 集成方式 |
|----------|----------|----------|
| `06 -> 00` | 事件总线、配置中心、安全工具、审计日志、缓存、国际化 | 统一接入基础设施能力 |
| `06 -> 01` | Person、Account、Assignment、IdentityContext | 解析接收人、身份快照、移动端身份刷新 |
| `02 -> 06` | `todo.*`、`process.*` 事件 | 生成待办提醒、超时提醒、流程结果通知 |
| `03 -> 06` | 内容发布与可见范围事件 | 生成公告与内容提醒 |
| `04 -> 06` | 未读消息数、最新消息列表、消息跳转卡片 | 门户消费消息读模型输出 |
| `05 -> 06` | 会议、日程、任务、合同、考勤等业务事件 | 统一生成业务提醒 |
| `06 -> 07` | 无直接主数据依赖 | 开放 API、Webhook、通用连接器与跨系统同步边界留在 `07` |

### 5.2 事件契约

本模块内部领域事件以 `msg.message.*`、`msg.channel.*` 命名；跨模块总线事件统一注册 `msg.notification.*`。

| 事件类型 | 触发时机 | 消费模块 |
|----------|----------|----------|
| `msg.notification.sent` | 渠道投递任务被系统接受；`channel=INBOX` 表示站内消息成功入箱 | 04/审计与监控 |
| `msg.notification.delivered` | 渠道确认送达 | 04/审计与监控 |
| `msg.notification.read` | 用户已读站内消息 | 04 |
| `todo.item.created` | 入站业务事件，生成待办提醒 | 06 消费 |
| `todo.item.overdue` | 入站业务事件，生成催办/升级提醒 | 06 消费 |
| `process.task.overdue` | 入站业务事件，生成超时提醒 | 06 消费 |
| `org.account.locked` | 入站业务事件，生成安全告警 | 06 消费 |
| `org.identity.switched` | 入站业务事件，刷新未读数和身份相关缓存 | 06 消费 |
| `org.identity-context.invalidated` | 入站业务事件，失效身份相关缓存，并按回退身份重建或强制退出移动会话 | 06 消费 |

### 5.3 待办与消息分工（D16）

| 视图 | 职责 | 管理模块 |
|------|------|----------|
| 待办视图 | 事务处理入口，待办/已办/我发起/抄送等处理视图 | `02-流程与表单` |
| 消息视图 | 通知触达入口，站内消息、外部渠道提醒、移动端未读聚合 | `06-消息移动与生态` |

**联动规则**：

- `todo.item.created` -> 生成消息提醒
- `todo.item.overdue` / `process.task.overdue` -> 生成催办或升级提醒
- 待办完成 -> 可将相关消息标记为已失效或归档，但不代表消息“已读”
- 消息已读 -> 不回写改变待办处理状态

### 5.4 与开放能力的边界（D04）

| 能力 | 归属 | 说明 |
|------|------|------|
| 用户触达、消息路由、移动端入口、生态登录接入 | `06` | 面向用户与终端设备的触达和接入能力 |
| 开放 API、Webhook、通用连接器、跨系统数据交换与服务治理 | `07` | 面向第三方系统和通用集成的开放能力 |

## 6. 身份上下文与移动端集成（D08）

### 6.1 身份上下文引用原则

- 接收人解析、目标任职、目标岗位统一引用 `01` 输出，不在 `06` 内重建组织与授权模型。
- 对待办提醒、身份相关提醒等场景，`Notification` 必须快照 `targetAssignmentId` 与 `targetPositionId`。
- 身份切换后，消息未读数、列表过滤条件、移动端首页角标和门户消息卡片必须同步刷新。
- 已生成消息保留“生成时语义”，不因后续身份切换而重新归属。

### 6.2 设备与会话集成规则

- `DeviceBinding` 负责设备可信度、Push Token 和绑定状态，不负责身份授权计算。
- `MobileSession` 负责设备级会话续期和风险冻结，当前岗位/任职仅作为移动端快速判定快照。
- 设备丢失、撤销或高风险冻结时，应同时失效对应会话和主动触达能力。

## 7. 运行治理策略

### 7.1 订阅治理

- 订阅规则必须只引用统一事件契约中的事件类型和载荷字段。
- 用户订阅偏好、静默规则和升级策略必须支持审计、回滚和差异对比。
- 对重复事件或高频事件必须通过去重键和时间窗抑制重复提醒。

### 7.2 投递治理

- `DeliveryTask` 失败时按统一退避策略重试，超过上限进入人工补偿队列。
- 站内入箱与外部投递分离治理，外部失败不得污染站内消息事实。
- 模板渲染结果和渠道响应必须保留快照，支撑送达追踪与问题排查。

### 7.3 移动治理

- 设备绑定、Push Token、移动会话都必须支持快速失效和风险冻结。
- 高频接口以轻量聚合和缓存为主，避免移动端直接扇出多个业务服务。
- 弱网场景下优先保障消息列表、待办入口、审批处理和阅读预览可恢复。

### 7.4 生态治理

- 企业微信、钉钉、SSO、LDAP、短信、邮件等接入必须统一做连接测试、状态监控和异常日志留存。
- 回调处理必须完成签名校验、租户隔离和幂等处理。
- 接入配置变更、连接异常、手动补偿都必须进入统一审计。

## 9. 一期交付范围

### 9.1 一期包含

- `message-center`：统一消息模型、消息列表、已读/归档/撤回状态管理
- `channel-sender`：基础模板、渠道端点、路由策略、送达追踪、失败重试与人工补偿入口
- `event-subscription`：业务事件接入、订阅规则、订阅偏好、静默规则、优先级升级
- `mobile-support`：设备绑定、移动会话、消息与待办高频接口、弱网容错基础能力
- `ecosystem`：至少一种生态渠道或登录方式接入、连接测试、状态监控和回调审计

### 9.2 一期简化

- 模板能力先支持基础变量替换和模板预览，不做复杂可视化模板设计器。
- 路由策略先支持固定渠道顺序、静默规则和简单升级策略，不做复杂条件编排。
- 事件接入先聚焦 `todo.*`、`process.*` 等主链路事件，内容和更多业务提醒按阶段接入。
- 移动端先聚焦消息、待办、审批、阅读、预览等高频场景，不复制完整 PC 工作台。
- 生态接入先按项目打通一种渠道或登录方式，不追求一次性覆盖全部生态矩阵。

### 9.3 一期暂缓

- 周期 Digest、批量摘要、节假日静默等复杂订阅策略
- 多级升级链路、复杂路由编排和更细粒度触达效果分析
- 全量生态矩阵接入和复杂离线补偿/命令队列能力

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 待办与消息边界混淆，导致状态互相污染 | 严格区分事务处理真相与通知触达真相，联动只通过事件与状态映射 |
| 外部渠道失败影响用户站内提醒 | 站内入箱与外部投递分离建模，外部失败不回滚站内消息 |
| 高优先级消息重复轰炸用户 | 通过订阅规则、去重时间窗、静默策略和升级策略统一治理 |
| 身份切换后消息未读数或过滤条件不一致 | 所有身份相关消息统一快照岗位/任职，并在切换后统一刷新缓存和读模型 |
| 生态接入配置和回调问题难排查 | 统一连接测试、状态监控、回调审计和人工补偿入口 |
