# channel-sender 概览

## 1. 模块定位

`channel-sender` 是 `06-消息移动与生态` 中负责“怎么发、往哪发、发得怎么样”的子模块，持有 `MessageTemplate`、`ChannelEndpoint`、`RoutingPolicy`、`DeliveryTask`、`DeliveryAttempt`。

该模块负责把已生成的消息快照路由为站内 `INBOX` 和可选外部渠道投递任务，记录送达过程、失败重试和人工补偿入口，但不持有消息已读事实。

## 2. 核心职责

| 职责 | 说明 |
|------|------|
| 模板管理 | 管理模板编码、版本、语言、多渠道渲染定义 |
| 渠道端点管理 | 管理 `INBOX / EMAIL / SMS / WECHAT_WORK / DINGTALK / APP_PUSH` 端点及其配置引用 |
| 路由策略 | 根据类别、优先级、静默规则和降级链路决定发往哪些渠道 |
| 投递治理 | 创建 `DeliveryTask`、记录 `DeliveryAttempt`、处理重试、补偿和送达汇总 |

## 3. 上下游边界

### 3.1 上游输入

- `message-center` 提供已落库的 `Notification` 及其消息快照。
- `event-subscription` 提供已解析的类别、优先级、模板编码和业务上下文。
- `ecosystem` 提供外部渠道与登录接入的配置引用和健康状态。

### 3.2 下游输出

- 对 `message-center` 回写投递汇总状态：`PENDING / PARTIAL_SUCCESS / SUCCESS / FAILED`。
- 对门户、管理端提供送达状态、失败原因和人工重试入口。
- 对外部渠道适配器提供标准化投递任务输入。

### 3.3 明确不负责

- 不负责消息列表、消息详情、已读事实和身份快照管理。
- 不负责订阅规则、用户静默偏好、事件接入。
- 不负责移动设备绑定与会话。
- 不负责第三方连接配置主数据，只引用 `ecosystem` 暴露的 `configRef` 与健康状态。

## 4. 核心能力表

| 能力 | 输入 | 输出 | 关键规则 | 一期约束 |
|------|------|------|----------|----------|
| 模板版本管理 | 模板编码、渠道、语言、变量定义 | 已发布模板版本 | 已发布版本不可原地修改 | 仅支持基础变量替换和预览 |
| 渠道端点治理 | 渠道类型、配置引用、限流与配额 | 可用端点清单 | 不保存明文密钥；`INBOX` 为内建端点 | 至少保证 `INBOX` 可用 |
| 路由与降级 | 类别、优先级、用户偏好、健康状态 | 渠道顺序、降级链路 | 一期所有消息至少进入 `INBOX` | 只支持固定顺序和简单升级 |
| 投递追踪与补偿 | 投递任务、重试结果、回执 | 投递状态、失败原因、补偿入口 | 外部失败不得回滚站内入箱 | 统一退避 `1s -> 5s -> 30s -> 5m -> 30m` |

## 5. 典型业务流

### 5.1 多渠道发送

1. `message-center` 发布 `msg.message.created`。
2. 本模块读取消息快照和模板编码，匹配 `RoutingPolicy`。
3. 至少生成一条 `INBOX` 投递任务，必要时补充邮件、短信、企微、钉钉或 App Push 任务。
4. 每条任务独立执行并记录 `DeliveryAttempt`。
5. 外部渠道成功时更新 `deliveredAt`；失败时进入统一退避重试。
6. 超过重试上限进入 `GAVE_UP`，暴露人工补偿入口。
7. 汇总每条消息的投递结果回写 `message-center.deliveryStatus`。

### 5.2 路由降级

1. 首选端点不可用或配额不足。
2. 根据 `fallbackChannelOrder` 降级到次选外部渠道。
3. 无论外部链路如何变化，`INBOX` 入箱不受影响。

## 6. 一期实现约束

- 所有消息必须至少进入 `INBOX`；`INBOX` 为内建逻辑端点，不允许禁用。
- 模板只支持显式变量定义、基础文本替换和预览，不做可视化编排器。
- 路由策略只支持固定渠道顺序、静默规则和简单升级，不做复杂条件表达式。
- 外部渠道失败重试统一采用固定退避序列，达到上限后必须有人工补偿入口。
- 端点配置只保存 `configRef`，所有密钥、密码、令牌由配置中心或密钥库管理。
- `channel-sender` 只拥有投递事实，不拥有第三方平台配置主数据。
