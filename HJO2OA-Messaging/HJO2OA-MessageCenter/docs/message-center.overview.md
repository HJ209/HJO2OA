# message-center 概览

## 1. 模块定位

`message-center` 是 `06-消息移动与生态` 中面向用户通知真相的子模块，负责持有 `Notification` 与 `NotificationAction`，统一管理站内消息入箱、状态流转、身份快照和消息查询视图。

该模块只负责“消息是什么、给谁、当前是否已读/归档/撤回/过期”，不负责模板渲染、外部渠道投递、订阅规则、移动会话或生态接入配置。

## 2. 核心职责

| 职责 | 说明 |
|------|------|
| 统一消息入箱 | 以“单接收人单消息”为粒度创建 `Notification`，保存标题、摘要、来源事件、深链和身份快照 |
| 消息状态管理 | 管理 `UNREAD / READ / ARCHIVED / REVOKED / EXPIRED` 状态，不做物理删除 |
| 查询与汇总 | 提供列表、详情、未读数、门户卡片和移动端所需的只读视图 |
| 动作留痕 | 对已读、归档、撤回、过期等动作记录 `NotificationAction` 审计轨迹 |

## 3. 上下游边界

### 3.1 上游输入

- `event-subscription` 根据业务事件命中订阅规则后，向本模块发起内部建消息命令。
- `01-identity-context` 提供 `Person / Assignment / Position` 真相，用于接收人和身份快照写入。
- `channel-sender` 回写投递汇总状态，但不修改消息已读事实。

### 3.2 下游输出

- `04-门户与工作台` 消费未读数、最新消息列表和深链跳转信息。
- `mobile-support` 消费移动端消息列表、未读数和身份相关筛选结果。
- `channel-sender` 以已创建消息为输入，生成 `INBOX` 和外部渠道投递任务。

### 3.3 明确不负责

- 不负责模板、变量渲染、渠道端点、路由策略和送达重试，这些由 `channel-sender` 承担。
- 不负责订阅规则、静默策略、优先级升级，这些由 `event-subscription` 承担。
- 不负责设备绑定、移动会话和弱网容错，这些由 `mobile-support` 承担。
- 不负责企业微信、钉钉、SSO、LDAP、短信、邮件接入配置，这些由 `ecosystem` 承担。

## 4. 核心能力表

| 能力 | 输入 | 输出 | 关键规则 | 一期约束 |
|------|------|------|----------|----------|
| 消息创建 | 业务事件快照、接收人、模板结果、深链 | `Notification` | 单接收人单消息；支持 `dedupKey` 去重 | 仅内部命令写入，不对终端开放创建接口 |
| 列表与详情 | 人员身份、筛选条件、分页参数 | 消息列表、消息详情 | 仅返回当前人员可见消息；任职相关消息按快照过滤 | 支持类别、状态、来源模块、时间范围筛选 |
| 状态流转 | 已读、归档、撤回、过期命令 | 状态更新、动作轨迹 | 不物理删除；撤回和过期后仍保留审计 | 支持单条与批量已读，归档/撤回保留原因 |
| 未读汇总 | 人员、任职上下文 | 总未读数、分类型未读数 | 身份切换后按快照重新计算可见范围 | 门户和移动端共用口径 |

## 5. 典型业务流

### 5.1 待办提醒入箱

1. `todo.item.created` 被 `event-subscription` 消费并命中订阅规则。
2. `event-subscription` 解析接收人、任职快照、优先级和 `dedupKey`。
3. `message-center` 创建 `Notification`，状态初始化为 `UNREAD`，投递状态初始化为 `PENDING`。
4. `channel-sender` 基于该消息创建 `INBOX` 和可选外部渠道投递任务。
5. 用户在门户或移动端查看列表并打开详情。
6. 用户执行“已读”后，本模块记录 `NotificationAction(READ)` 并更新未读数。

### 5.2 消息失效与归档

1. 上游业务完成待办或消息过期条件命中。
2. 内部命令触发消息状态变更为 `EXPIRED` 或 `ARCHIVED`。
3. 已存在的读状态不回写业务待办事实，消息只作为通知留痕。

## 6. 一期实现约束

- 所有消息至少生成一条站内 `INBOX` 入箱事实，外部渠道失败不得回滚站内入箱。
- 消息必须快照 `targetAssignmentId`、`targetPositionId`，消息可见性以“生成时语义”为准。
- `deepLink` 只能指向门户页、移动页或业务详情入口，不得暴露内部表结构。
- 已读不代表待办已处理；待办完成也不自动等价为消息已读。
- 不支持群体共享消息主记录；群发场景由应用层拆分为多条 `Notification`。
- 不支持物理删除、回收站和复杂消息编辑；撤回与过期均保留审计轨迹。
