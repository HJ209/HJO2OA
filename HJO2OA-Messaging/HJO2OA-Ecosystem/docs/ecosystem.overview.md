# ecosystem 概览

## 1. 模块定位

`ecosystem` 是 `06-消息移动与生态` 中负责第三方生态接入配置、回调校验、健康状态治理和接入审计的子模块，持有 `EcosystemIntegration` 与 `CallbackAuditRecord`。

该模块面向企业微信、钉钉、短信、邮件、SSO、LDAP 等生态对象提供连接治理能力；它不拥有消息投递事实，也不承担通用开放接口和跨系统同步职责。

## 2. 核心职责

| 职责 | 说明 |
|------|------|
| 接入配置管理 | 管理生态接入类型、认证方式、配置引用和启停状态 |
| 回调校验与审计 | 处理第三方回调、签名校验、租户隔离和审计留痕 |
| 健康监控 | 维护接入健康状态、最近检查时间和错误摘要 |
| 接入排障 | 提供连接测试、异常排查和状态展示能力 |

## 3. 上下游边界

### 3.1 上游输入

- `channel-sender` 引用本模块维护的 `configRef` 与健康状态，发起外部渠道投递。
- 统一认证或生态登录流程引用本模块的 SSO/LDAP 配置。
- 第三方平台回调请求进入本模块完成签名和租户校验。

### 3.2 下游输出

- 向管理端提供接入配置、状态监控、回调审计和连接测试结果。
- 向 `channel-sender` 暴露端点可用性所需的只读治理信息。

### 3.3 明确不负责

- 不负责消息模板、路由策略、投递任务和送达重试，这些由 `channel-sender` 负责。
- 不负责通用开放 API、Webhook 平台、跨系统数据同步或连接器治理，这些由 `07-数据服务与集成` 负责。
- 不负责组织主数据同步的最终落表，只负责 LDAP/SSO 接入配置与认证入口。

## 4. 核心能力表

| 能力 | 输入 | 输出 | 关键规则 | 一期约束 |
|------|------|------|----------|----------|
| 接入配置管理 | 集成类型、认证方式、配置引用 | `EcosystemIntegration` | 只存 `configRef`，不存明文密钥 | 至少打通一种生态渠道或登录方式 |
| 回调校验 | 第三方请求、签名、时间戳、租户线索 | 校验结果与审计记录 | 必须先校验签名和租户，再进入业务处理 | 只支持父模块定义的接入类型 |
| 健康监控 | 定时检测、连接测试结果、失败摘要 | `healthStatus`、`lastCheckAt` | 健康状态只影响可用性判断，不自动改写业务事实 | 必须支持手工测试 |
| 排障与审计 | 回调记录、连接错误、测试结果 | 审计日志和状态页 | 所有接入变更和异常都要留痕 | 不做复杂自动自愈 |

## 5. 典型业务流

### 5.1 新接入启用

1. 管理员创建一条 `EcosystemIntegration`。
2. 配置认证方式、回调地址、签名算法和 `configRef`。
3. 发起连接测试，确认 `healthStatus=HEALTHY`。
4. `channel-sender` 在端点侧引用该 `configRef`，开始外部渠道投递。

### 5.2 回调处理

1. 第三方平台调用回调入口。
2. 本模块校验签名、租户和回调类型。
3. 记录 `CallbackAuditRecord`。
4. 将合法结果交由发送链路或认证链路消费，非法请求直接拒绝。

## 6. 一期实现约束

- 一期至少支持一种生态渠道或登录方式落地，不追求全量生态矩阵。
- 接入配置只保存 `configRef`，所有密钥、密码、证书由外部配置中心或密钥库管理。
- 生态回调必须完成签名校验、租户隔离和幂等处理。
- 本模块默认不发布独立领域事件，可消费投递链路事件做适配或状态治理。
- 不承载通用连接器、开放接口平台和跨系统数据同步调度。
