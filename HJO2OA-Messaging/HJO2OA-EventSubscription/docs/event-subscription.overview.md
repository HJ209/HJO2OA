# event-subscription 概览

## 1. 模块定位

`event-subscription` 是 `06-消息移动与生态` 中负责“哪些业务事件要转成什么提醒、提醒给谁”的子模块，持有 `SubscriptionRule` 与 `SubscriptionPreference`。

该模块以消费业务事件为主，通过规则匹配、目标解析、静默与升级策略，产出消息创建命令；它不持有消息正文真相，也不持有渠道投递事实。

## 2. 核心职责

| 职责 | 说明 |
|------|------|
| 事件接入 | 只消费统一事件契约中已注册的业务事件 |
| 订阅规则管理 | 维护事件类型、目标解析方式、模板编码、默认优先级和去重规则 |
| 用户偏好管理 | 管理允许渠道、静默时段、摘要模式和升级开关 |
| 提醒生成治理 | 计算提醒频率、优先级升级和去重，向下游发起消息创建命令 |

## 3. 上下游边界

### 3.1 上游输入

- 业务模块发布的统一事件，重点包括 `todo.*`、`process.*`、`org.account.locked`。
- `01-identity-context` 提供人员、任职、岗位真相，用于目标解析。

### 3.2 下游输出

- 向 `message-center` 发起内部消息创建命令。
- 向管理端提供规则、偏好、执行审计视图。

### 3.3 明确不负责

- 不负责通用 Webhook、开放 API、跨系统事件接入，相关能力归 `07-数据服务与集成`。
- 不负责消息列表、消息状态、渠道投递、设备会话或生态配置。
- 一期不对总线发布独立领域事件。

## 4. 核心能力表

| 能力 | 输入 | 输出 | 关键规则 | 一期约束 |
|------|------|------|----------|----------|
| 事件匹配 | 已注册业务事件 | 命中的规则集合 | 只允许引用统一事件契约字段 | 一期聚焦 `todo.*`、`process.task.overdue`、`org.account.locked` |
| 目标解析 | 事件载荷、组织身份数据 | 接收人、任职快照 | 待办类优先解析 `assignmentId`、`positionId` | 仅支持父文档定义的解析方式 |
| 偏好与静默 | 用户偏好、默认策略 | 渠道覆盖、静默结果 | 强制消息不可被完全静默 | 不支持复杂节假日规则 |
| 去重与升级 | `dedupKey`、优先级、频率策略 | 单次提醒命令 | 避免重复轰炸；允许高优先级升级 | 不支持复杂 digest 编排 |

## 5. 典型业务流

### 5.1 待办创建提醒

1. 接收到 `todo.item.created`。
2. 根据 `SubscriptionRule` 匹配消息类别、模板编码和默认优先级。
3. 解析接收人及其 `assignmentId`、`positionId`。
4. 应用 `SubscriptionPreference`，计算允许渠道、静默窗口和是否允许升级。
5. 生成 `dedupKey`，校验是否在去重窗口内重复触发。
6. 向 `message-center` 发起内部消息创建命令。

### 5.2 超时升级提醒

1. 接收到 `todo.item.overdue` 或 `process.task.overdue`。
2. 命中升级规则，将优先级提升到 `URGENT` 或 `CRITICAL`。
3. 如果用户允许升级，交由 `channel-sender` 决定更强外部触达链路。

## 6. 一期实现约束

- 只消费统一事件契约中已注册的事件，不接入发布方私有模型。
- 一期主链路仅保证 `todo.item.created`、`todo.item.overdue`、`process.task.overdue`、`org.account.locked`。
- `SubscriptionPreference` 只能覆盖非强制消息；安全类、账号锁定类消息不可被完全静默。
- 不发布独立领域事件，输出统一为内部消息创建命令。
- 不实现复杂 digest、节假日静默、可编排表达式引擎。
