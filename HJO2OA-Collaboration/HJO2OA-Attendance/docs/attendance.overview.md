# attendance 模块概览

## 模块定位

| 项目 | 说明 |
|------|------|
| 模块定位 | 承载 `AttendanceRequest` 聚合，负责请假、出差、加班、补卡、外出等申请型考勤台账及结果回写。 |
| 领域对象 | `AttendanceRequest`、`AttendanceEvidence`、规则快照引用。 |
| 一期范围 | 申请录入、审批接入、规则快照冻结、结果回写、异常与结果查询。 |
| 不负责内容 | 不建设高级排班引擎、不接入打卡硬件、不自建消息中心、不复制审批能力。 |

## 核心职责

| 职责 | 说明 | 边界口径 |
|------|------|----------|
| 申请台账管理 | 维护申请类型、申请人、时间区间、时长、状态和结果状态。 | 考勤业务台账归 `05`。 |
| 规则快照冻结 | 提交申请时冻结班次/规则快照，保证历史申请不受规则变更影响。 | 复杂排班与算法不在一期实现。 |
| 审批结果回写 | 接收 `02` 流程结果，显式回写为 `APPROVED/REJECTED/RESULTED` 等业务状态。 | 审批实例与待办仍归 `02`。 |
| 佐证与异常处理 | 管理附件佐证、异常标记和补偿结果。 | 附件存储统一复用 `00`。 |

## 上下游边界

| 方向 | 模块 | 边界说明 |
|------|------|----------|
| 上游 | `01` 身份与权限 | 提供申请人、审批责任人、组织可见范围和 HR 管理权限。 |
| 上游 | `02` 流程审批 | 提供申请审批、终止回调和审批轨迹。 |
| 上游 | `00` 附件中心 | 提供请假证明、车票、定位截图等佐证附件引用。 |
| 下游 | `06` 消息与触达 | 消费考勤申请与结果变化事件进行消息提醒。 |
| 下游 | `04` 门户与办公中心 | 展示我的申请、异常待处理、结果统计卡片。 |
| 下游 | `07` 开放集成 | 对外同步考勤结果与统计统一经 `07` 暴露。 |

## 核心能力

| 能力 | 说明 | 一期口径 |
|------|------|----------|
| 多类型考勤申请 | 支持请假、出差、加班、补卡、外出五类申请。 | 统一收敛到 `AttendanceRequest`。 |
| 规则快照与结果回写 | 提交时固化规则快照，审批完成后写入结果状态。 | 先支持基础规则，不做高级排班。 |
| 佐证附件管理 | 绑定多份证明、票据、定位等附件。 | 仅存附件引用与类型。 |
| 异常与结果查询 | 提供个人/组织维度的申请结果和异常视图。 | 统计口径以已回写结果为准。 |

## 典型业务流

| 业务流 | 步骤 |
|--------|------|
| 请假申请 | 新建申请 -> 冻结规则快照 -> 绑定佐证 -> 提交 `02` 审批 -> 发布 `biz.attendance.requested`。 |
| 审批回写结果 | 接收 `process.instance.completed` -> 回写 `APPROVED/RESULTED` -> 计算结果状态 -> 发布 `biz.attendance.result-changed`。 |
| 补卡补偿 | 员工提交补卡 -> 审批通过 -> 写回异常补偿结果 -> 门户与消息刷新。 |

## 一期实现约束

| 约束 | 说明 |
|------|------|
| 审批统一复用 `02` | 本模块不自建审批流程、待办中心和审批意见存储。 |
| 排班能力简化 | 一期只支持基础规则快照，不做高级排班、跨班次复杂计算。 |
| 附件统一复用 `00` | 仅保存佐证附件引用，不复制附件实体。 |
| 结果显式回写 | 流程通过后必须回写结果状态，不能仅依赖流程完成事实。 |
