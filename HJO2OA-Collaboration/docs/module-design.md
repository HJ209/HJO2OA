# 05-协同办公应用 模块设计

## 1. 文档目的

本文档定义 `05-协同办公应用` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围。作为模块功能文档、领域模型文档和开发任务文档的上层约束文档。

对应架构决策编号：D03（`03` 与 `05` 模块边界）、D12（事件契约）、D14（内容底座约束）、D16（待办与消息分工）、D20（一期实施范围）、ADR-006（参考设计策略）。

## 2. 设计策略

### 2.1 参考设计原则

本模块参考 O2OA 内置公文、会议、任务、考勤、合同、资产等业务应用的能力闭环，但不沿用其业务与底座深耦合的实现方式（ADR-006）。具体原则：

- **借鉴高频办公应用组合**：以公文、会议、日程、任务、考勤等高频场景作为业务装配中心。
- **不借鉴底座重复建设**：审批、消息、内容、附件、开放接口等能力统一复用平台模块，不在 `05` 内重复实现。
- **不借鉴业务孤岛式应用**：各业务应用通过统一事件、统一入口和统一状态联动形成协同，而不是各自封闭。
- **优先业务语义清晰，再做跨应用联动**：先固化各业务台账和状态机，再叠加统一入口、门户卡片和移动端体验。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| `05` 拥有业务语义和台账规则 | 公文、会议、任务、考勤、合同、资产状态和规则归 `05` |
| 审批能力统一复用 `02` | 业务应用不内嵌自有流程引擎或待办中心 |
| 内容型场景统一复用 `03` | 公告、知识、共享文件目录不再在 `05` 自建内容底层模型 |
| 消息触达统一复用 `06` | 业务应用只发布事件和提醒意图，不直接管理多渠道送达 |
| 门户入口统一复用 `04` | `05` 输出业务卡片数据和跳转协议，不编排首页模板 |
| 对外集成统一复用 `07` | 业务应用不自建开放 API、连接器和同步治理 |
| `cross-app-linkage` 只做编排不做持有 | 统一入口、跳转和状态联动不形成独立主数据层 |

### 2.3 O2OA 能力借鉴清单

| O2OA 能力 | HJO2OA 一期保留 | HJO2OA 一期简化 | HJO2OA 暂缓 |
|-----------|----------------|----------------|-------------|
| 公文收发文能力 | ✅ | - | - |
| 会议安排、签到、纪要 | ✅ | - | - |
| 日程与任务协同 | ✅ | - | - |
| 考勤申请与统计 | ✅ | 基础规则 | 高级排班 |
| 合同和资产台账 | ✅ | 基础台账 | 复杂财务/设备体系 |
| 公告与共享文件场景 | ✅ | 基于 `03/00` 装配 | 独立高级协作 |
| 跨应用统一入口 | ✅ | 基础入口和跳转 | 复杂编排引擎 |
| 移动端高频场景 | ✅ | 审批/会议/任务等核心页面 | 深度离线/生态融合 |

## 3. 分层架构

### 3.1 事务型办公业务层（Transactional Office Layer）

负责公文与会议两类强流程、强协同业务主场景。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `document-mgmt/` | 收发文、拟稿、核稿、签发、传阅、归档、签章记录 | `biz_document`、`biz_document_circulation`、`biz_document_seal_record` |
| `meeting-mgmt/` | 会议申请、排期、资源冲突校验、签到和纪要 | `biz_meeting`、`biz_meeting_participant`、`biz_meeting_sign_record`、`biz_meeting_minutes` |

**事务型办公层规则**：

- 公文和会议均可绑定 `02` 流程实例，但业务状态机独立于流程状态机。
- 会务资源冲突、文号生成、签章记录、纪要归档等规则由业务聚合自身维护。
- 所有关键状态变更必须产生可审计的业务事件。

### 3.2 执行协同层（Execution Collaboration Layer）

负责日程、任务和考勤等执行类业务。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `schedule-task/` | 日程安排、提醒规则、任务分配、延期反馈和完成闭环 | `biz_schedule_entry`、`biz_schedule_reminder_rule`、`biz_task_assignment`、`biz_task_feedback` |
| `attendance/` | 请假、出差、加班、补卡、外出申请与考勤结果 | `biz_attendance_request`、`biz_attendance_evidence` |

**执行协同层规则**：

- 任务是业务执行台账，不替代 `02` 的待办。
- 考勤申请的审批结果必须显式回写为考勤结果，而不是只保留流程完成记录。
- 日程和任务可以互相关联，但仍保持各自状态机独立。

### 3.3 台账与共享场景层（Ledger & Shared Scene Layer）

负责合同、资产和共享文件/公告等场景装配。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `contract-asset/` | 合同台账、履约提醒、资产台账和资产流转 | `biz_contract`、`biz_contract_milestone`、`biz_contract_reminder`、`biz_asset`、`biz_asset_transfer_record` |
| `bulletin-fileshare/` | 公告场景装配、共享空间、资源绑定、共享权限 | `biz_shared_space`、`biz_space_member`、`biz_space_resource_binding` |

**台账与共享场景层规则**：

- 合同和资产属于 `05` 自有台账主数据。
- 公告和知识类内容不归 `05` 持有，`bulletin-fileshare` 只装配 `03` 的已发布内容和 `00` 的附件。
- 共享空间只管理范围和资源绑定，不复制存储内容正文或附件二进制。

### 3.4 统一入口与编排层（Experience Orchestration Layer）

负责跨应用统一入口、跳转、状态联动和移动端高频体验。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `cross-app-linkage/` | 应用入口聚合、跨应用跳转、状态标识、移动端高频场景适配 | 无独立领域主表 |

**编排层规则**：

- 统一入口只消费业务聚合结果，不直接操作业务主表。
- 跨应用联动优先基于事件驱动和只读查询实现，不形成新的业务真相源。
- 移动端页面沿用各业务应用状态和权限判定，不单独创造第二套规则。

## 7. 业务编排策略

### 7.1 审批接入策略

- 需要审批的业务对象统一通过标准流程接入 `02`。
- 业务对象保留 `processInstanceId` 和关键审批摘要字段，避免把流程数据复制为业务主档。
- 流程完成或终止后，业务聚合必须自己决定是否生效、驳回、归档或取消。

### 7.2 内容与共享场景复用策略

- 公告场景直接消费 `03` 的已发布内容，不复制稿件、发布和检索模型。
- 共享文件场景通过 `SharedSpace + SpaceResourceBinding` 管理空间成员和资源入口。
- 文件预览和附件下载统一复用 `00 attachment` 能力。

### 7.3 门户与消息联动策略

- `04` 按业务事件和读模型消费会议、任务、合同提醒等数据，不改写业务状态。
- `06` 按 `biz.*` 事件生成站内信或外部渠道提醒。
- 移动端优先覆盖公文审批、会议查看、任务反馈和考勤申请等高频场景。

## 8. 一期交付范围

### 8.1 一期平台口径

- `05` 不属于平台最小闭环必做模块（D20）。
- 当前设计用于指导后续业务项目按需接入。

### 8.2 若项目提前接入，一期建议包含

- `document-mgmt`：收发文、拟稿、审批接入、传阅和归档
- `meeting-mgmt`：会议申请、排期、签到和纪要
- `schedule-task`：日程、提醒和任务闭环
- `attendance`：请假/出差/加班/补卡申请和结果回写

### 8.3 若项目提前接入，一期简化

- `contract-asset`：先支持基础台账、提醒和流转记录
- `bulletin-fileshare`：先支持公告装配、共享空间和资源绑定
- `cross-app-linkage`：先支持统一入口、跳转和状态标识

### 8.4 一期暂缓

- 复杂协作网盘能力（目录版本协作、多人编辑）
- 高级排班与复杂考勤计算引擎
- 合同与资产的深度财务联动、设备维保体系
- 复杂跨应用编排和智能工作台推荐

## 9. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 业务应用重复建设内容、消息、附件能力 | 强制复用 `03/06/00`，业务域只保留业务语义和状态 |
| 审批状态与业务状态混淆 | 业务聚合显式回写自身状态，流程状态仅作为审批事实 |
| `cross-app-linkage` 膨胀为新业务中台 | 限定其只做入口编排和跳转，不持有独立主数据 |
| 任务能力与待办中心职责重叠 | 明确任务是业务执行台账，待办是流程事务视图 |
| 共享空间演变为第二个内容底座 | 共享空间只管理范围和绑定，不存正文和附件主数据 |
