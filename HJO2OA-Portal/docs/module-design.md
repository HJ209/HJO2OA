# 04-门户与工作台 模块设计

## 1. 文档目的

本文档定义 `04-门户与工作台` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围。作为模块功能文档、领域模型文档、聚合读模型文档和开发任务文档的上层约束文档。

对应架构决策编号：D05（`04` 与 `07` 数据边界）、D08（身份上下文）、D15（门户聚合策略）、D20（一期实施范围）、ADR-006（参考设计策略）。

## 2. 设计策略

### 2.1 参考设计原则

本模块参考 O2OA 的门户首页、办公中心和门户设计器的能力闭环，但不沿用其产品结构与实现方式（ADR-006）。具体原则：

- **借鉴用户入口闭环**：统一首页、办公中心、角色化模板、个性化工作台、门户设计器等能力闭环直接借鉴思路。
- **不借鉴产品耦合结构**：O2OA 的门户、流程、消息、数据服务之间耦合较紧，HJO2OA 必须保持清晰边界。
- **不借鉴数据开放方式**：门户聚合只服务用户入口体验，不承担平台级开放接口和通用数据复用职责。
- **不借鉴页面拼接式实现**：门户首页与设计器应采用模板、区域、卡片、发布的结构化模型，而不是静态页面堆叠。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| 门户负责入口体验，不负责业务主数据 | 待办、消息、内容、会议、统计主数据均归所属业务域 |
| 聚合接口只服务门户与办公中心场景 | 不承载通用数据开放、第三方调用和系统集成 |
| 模板是设计时真相源，个性化是有限覆写 | 用户不能突破模板强约束或发布范围 |
| 设计器是工具层，不是领域真相源 | `portal-designer` 操作模板草稿，不单独持久化模板主数据 |
| 工作台卡片统一走聚合协议 | 前端不得绕过 `aggregation-api` 直接拼装多个业务接口 |
| 身份上下文驱动模板解析和快照隔离 | 身份切换后模板、权限、卡片快照必须同步重算 |

### 2.3 O2OA 能力借鉴清单

| O2OA 能力 | HJO2OA 一期保留 | HJO2OA 一期简化 | HJO2OA 暂缓 |
|-----------|----------------|----------------|-------------|
| 门户首页统一入口 | ✅ | - | - |
| 办公中心左右分栏工作区 | ✅ | - | - |
| 门户模板与页面配置 | ✅ | - | - |
| 卡片化聚合展示 | ✅ | - | - |
| 角色化/组织化门户 | ✅ | - | - |
| 用户个性化排序与快捷入口 | ✅ | 基础能力 | - |
| 门户设计器 | ✅ | 基础画布/属性编辑 | 高级脚本与资源编排 |
| 多端联动模板 | - | PC/移动基础支持 | 复杂多端协同 |
| 深度门户-报表一体化 | - | - | ✅ |
| 门户作为通用数据开放层 | - | - | ✅（由 `07` 承担） |

## 3. 分层架构

### 3.1 设计时（Design-Time）

负责门户模板、页面、区域、卡片定义与设计器配置。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `portal-model/` | 模板、页面、区域、卡片布置、版本链、发布草稿 | `portal_template`、`portal_template_version`、`portal_page`、`portal_layout_region`、`portal_widget_placement` |
| `widget-config/` | 卡片定义、数据源绑定、展示策略、卡片可见规则 | `portal_widget_def`、`portal_widget_data_source`、`portal_widget_display_policy` |
| `portal-designer/` | 设计器画布、模板编辑流程、预览触发 | 无独立领域主表 |

**设计时规则**：

- 门户模板采用版本链管理，已发布版本不可修改，只能基于新草稿版本调整。
- 页面结构、区域结构和卡片布置属于模板模型的一部分，不在设计器层重复建模。
- 卡片定义与页面布置解耦，卡片可以被多个模板、多个区域复用。
- 设计器只操作模板草稿，不直接改变当前线上生效模板。
- 模板必须显式声明布局模式：`THREE_SECTION`、`OFFICE_SPLIT`、`MOBILE_LIGHT` 或 `CUSTOM`。

### 3.2 发布与模板解析层（Publication & Resolution）

负责模板发布、生效时间、适用范围、优先级裁决和身份上下文下的模板解析。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `portal-model/` | 模板发布、发布时间窗、生效状态、范围绑定、优先级裁决 | `portal_publication`、`portal_publication_audience` |
| `personalization/` | 基于已生效发布模板的用户/任职级个性化覆写 | `portal_personalization_profile`、`portal_quick_access_entry` |

**发布与解析规则**：

- 发布对象是 `PortalVersion`，不是模板抽象本身。
- 同一 `sceneType + clientType` 下，用户最终只能解析出一个有效发布模板。
- 解析顺序遵循 `PERSON > POSITION > ROLE > DEPARTMENT > ORGANIZATION > GLOBAL`。
- 模板解析完成后，才可叠加个性化配置。
- 个性化配置只允许有限覆写模板，不得复制生成新的模板真相源。
- 一期仅支持显式范围灰度发布，不做复杂百分比灰度。

### 3.3 读模型（Query Model）

负责门户首页与办公中心所需的聚合读模型输出。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `aggregation-api/` | 待办、消息、公告、统计、快捷入口等卡片聚合与缓存 | `portal_agg_snapshot`（缓存快照，可选落库） |

**读模型规则**：

- 聚合接口输出的是用户侧读模型，不是业务主数据。
- 所有聚合结果必须以当前身份上下文裁剪。
- 快照键至少包含 `tenantId + personId + assignmentId/positionId + sceneType + cardType`。
- 一期默认采用 Redis 级缓存和异步失效策略；保留多级缓存扩展位，但不强制启用 L1 进程缓存。
- 聚合接口支持批量获取和单卡片刷新，前端不得绕过聚合层直接拼装数据。

### 3.4 渲染与交互层（Rendering & Interaction）

负责门户首页、办公中心、移动工作台和设计器交互。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `portal-home/` | 模板渲染、办公中心布局、卡片跳转、移动端工作台展示 | 无独立后端持久化 |
| `portal-designer/` | 门户编辑画布、属性编辑、模板预览、多端预览 | 无独立后端持久化 |

**渲染层规则**：

- 渲染层只消费模板解析结果、个性化配置和聚合快照。
- 首页布局和办公中心布局使用同一门户模板体系，不分裂为两套独立配置模型。
- 移动工作台消费轻量化模板和聚合协议，不重复建设独立配置中心。
- 一期不依赖 WebSocket 实时推送刷新，采用前端轮询 + 操作后定向刷新。

### 5.2 事件契约

本模块父级事件清单以 Portal 各子模块 `events/*.events.md` 为准；涉及统一总线对外暴露时，命名仍遵循 `docs/contracts/unified-event-contract.md`，模块前缀为 `portal`。当前在 Portal 域内真实发布并传播的核心事件如下：

| 事件类型 | 触发时机 | 主要消费方 |
|----------|----------|------------|
| `portal.template.created` | 新模板创建完成 | `portal-designer` 模板目录与版本链初始化 |
| `portal.template.published` | 模板版本发布成功 | `portal-designer` 版本时间线与模板基线刷新 |
| `portal.template.deprecated` | 历史模板版本废弃 | `portal-designer` 版本状态与预览基线刷新 |
| `portal.publication.activated` | 发布进入生效状态 | `personalization`、`aggregation-api`、`portal-home`、`portal-designer` |
| `portal.publication.offlined` | 发布下线或失效 | `personalization`、`aggregation-api`、`portal-home`、`portal-designer` |
| `portal.widget.updated` | 卡片定义、数据协议或展示规则更新 | `portal-model`、`aggregation-api`、`portal-designer` |
| `portal.widget.disabled` | 卡片停用 | `portal-model`、`aggregation-api`、`portal-designer` |
| `portal.personalization.saved` | 个性化保存成功 | `aggregation-api`、`portal-home` |
| `portal.personalization.reset` | 个性化重置成功 | `aggregation-api`、`portal-home` |
| `portal.snapshot.refreshed` | 聚合卡片快照刷新成功 | `portal-home` |
| `portal.snapshot.failed` | 聚合卡片快照刷新失败 | `portal-home`、诊断与降级展示逻辑 |

### 5.3 入站事件与刷新策略

| 事件类型 | 处理方式 | 刷新范围 |
|----------|----------|----------|
| `todo.item.created` / `todo.item.completed` / `todo.item.overdue` | 失效待办类快照 | 当前用户待办卡片 |
| `msg.notification.sent` / `msg.notification.read` | 失效消息类快照 | 当前用户消息卡片 |
| `content.article.published` | 失效内容类快照 | 受影响可见范围内用户的内容卡片 |
| `org.identity.switched` | 全量失效当前用户门户快照 | 当前用户全部卡片 |
| `org.identity-context.invalidated` | 全量失效当前用户门户快照，并按 `fallbackAssignmentId` 重建或触发重新登录 | 当前用户全部卡片 |
| `org.role.position-bound` / `org.role.person-granted` | 失效模板解析缓存与卡片可见范围缓存 | 受影响用户 |
| `process.instance.started` / `process.instance.completed` | 刷新基础流程统计和常用流程卡片 | 流程统计卡片 |
| `biz.meeting.created` / `biz.task.assigned` / `biz.task.completed` | 二期/三期刷新会议与任务卡片 | 对应业务卡片 |

### 5.4 门户聚合与开放数据的边界（D05）

| 场景 | 归属模块 | 说明 |
|------|----------|------|
| 用户登录后获取首页卡片数据 | `04` | 入口体验读模型 |
| 第三方系统通过 API 获取待办数量 | `07` | 通用开放接口能力 |
| 跨系统数据同步 | `07` | 数据同步与交换能力 |
| 门户图卡展示流程统计 | `04` 消费 `07/业务域` 输出 | `04` 只展示用户侧结果，不定义平台统计底座 |

### 5.5 门户、待办与消息分工

| 视图 | 职责 | 管理模块 |
|------|------|----------|
| 门户视图 | 统一入口、聚合卡片、工作台布局 | `04-门户与工作台` |
| 待办视图 | 事务处理入口、已办/我发起/抄送等事务列表 | `02-流程与表单` |
| 消息视图 | 通知触达入口、未读消息、送达状态 | `06-消息移动与生态` |

映射规则：

- 门户待办卡片读取 `02` 的读模型结果
- 门户消息卡片读取 `06` 的读模型结果
- 门户首页只是聚合入口，不回写改变待办或消息事实状态
- 卡片点击后跳转到待办中心或消息中心处理页面，而不是在门户层重实现业务逻辑

## 6. 身份上下文集成（D08）

| 阶段 | 集成方式 |
|------|----------|
| 登录后首次进入门户 | 根据 `X-Identity-Assignment-Id` 解析当前身份模板和聚合缓存 |
| 门户模板匹配 | 按组织/部门/岗位/角色/人员进行模板发布命中解析 |
| 聚合查询 | 当前身份决定待办、消息、内容和业务卡片可见范围 |
| 个性化读取 | 优先读取当前任职级个性化配置，回退到用户全局个性化 |
| 身份切换后 | 失效模板解析缓存、卡片快照缓存和个性化视图解析结果 |

设计要求：

- `assignmentId` 是门户模板解析和聚合快照隔离的关键主键之一。
- 个性化配置不得跨身份透传，避免兼岗看到主岗专属门户内容。
- 门户模板可见性与卡片可见性均以 `01-组织与权限` 输出的身份上下文为准。

## 7. 聚合缓存与刷新策略（D15）

### 7.1 缓存层次

目标架构支持三层：

| 层级 | 存储 | 说明 |
|------|------|------|
| L1 | Caffeine | 进程内高频热点缓存 |
| L2 | Redis | 跨实例共享缓存与失效通知 |
| L3 | 上游业务读模型 | 缓存未命中时回源 |

一期启用策略：

- 启用 `L2 Redis + L3 回源`
- 暂不强制启用 L1 进程缓存
- 暂不强制启用 WebSocket 实时刷新

### 7.2 Key 设计

推荐快照 Key：

```text
portal:agg:{tenantId}:{personId}:{assignmentId}:{sceneType}:{cardType}
```

推荐模板解析 Key：

```text
portal:publication:{tenantId}:{personId}:{assignmentId}:{sceneType}:{clientType}
```

### 7.3 刷新原则

- 操作后定向刷新优先于全量首页刷新。
- 身份切换后必须整体失效当前用户快照。
- 快照刷新失败不能阻塞整页渲染，允许局部卡片降级。
- 快照返回允许 `STALE` 语义，即返回旧数据并异步刷新。

## 9. 一期交付范围

### 9.1 一期包含

- `portal-model`：模板、页面、区域、卡片布置、版本管理、模板发布、范围控制
- `widget-config`：基础卡片类型、数据源绑定、展示协议、卡片可见性配置
- `aggregation-api`：待办、消息、快捷入口、基础流程统计卡片的聚合输出
- `portal-home`：统一工作台首页与办公中心基础布局
- `personalization`：基础卡片排序、快捷入口、主题偏好
- `portal-designer`：基础门户设计器、草稿保存、发布预览

### 9.2 一期简化

- 内容聚合：一期由 `04` 内置简化公告能力提供，不依赖完整 `03` 内容底座
- 日程/会议聚合：一期暂不实现完整业务卡片
- 统计卡片：一期仅实现基础流程统计，不依赖 `02-process-monitor`
- 个性化：一期仅支持排序、隐藏、快捷入口和主题，不做复杂模板分叉
- 设计器：一期提供基础布局与属性编辑，不做脚本、资源文件、复杂多端联动
- 刷新机制：一期使用轮询 + 定向刷新，不强制实时推送

### 9.3 一期暂缓

- 复杂专题门户、多级导航体系
- 高级灰度发布与模板 AB 实验
- 复杂统计图卡与 `07-report` 深度联动
- 多品牌/多皮肤/多租户主题系统
- 高级设计器能力（脚本、资源文件、复杂组件编排）

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 门户逐渐演化成“万能接口层” | 强制 `04` 只提供用户侧聚合读模型，开放能力全部归 `07` |
| 前端绕过聚合层直接拉多源接口 | 统一卡片协议与聚合接口，前端禁止自拼多源数据 |
| 个性化配置膨胀为模板分叉 | 个性化只允许有限覆写，模板真相源始终归 `portal-model` |
| 身份切换后数据串用 | 缓存 Key 强制带 `assignmentId/positionId`，切换后全量失效 |
| 首页聚合性能不稳定 | 引入聚合快照缓存、定向失效和卡片级降级策略 |
| 一期内容/业务模块未齐全导致门户卡片不完整 | 采用一期简化策略，由 `04` 提供最小可用公告和快捷入口能力 |
