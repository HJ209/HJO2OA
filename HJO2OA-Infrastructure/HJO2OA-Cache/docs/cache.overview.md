# 缓存管理概览

## 模块定位

| 项 | 说明 |
|----|------|
| 模块角色 | `HJO2OA-Cache` 是 `00-平台基础设施` 执行与运行层 + 运维治理层子模块，负责缓存命名空间、失效轨迹和治理入口。 |
| 策略对象 | `CachePolicy` 定义命名空间、TTL、淘汰策略和失效方式。 |
| 运行对象 | `CacheInvalidationRecord`、命中率指标、热点统计属于运行对象。 |
| 边界强调 | 缓存不是事实真相源；缓存值随时可丢失、可重建，真相源仍在业务域或共享主数据域。 |

## 核心职责

| 职责 | 说明 |
|------|------|
| 命名空间治理 | 维护命名空间、后端类型、TTL、容量和淘汰策略。 |
| 失效编排 | 支持事件驱动、手工治理和到期淘汰三类失效路径，并记录失效轨迹。 |
| 运行监控 | 提供命中率、容量、热点 Key 和失效频次等基础指标。 |
| 治理入口 | 为运维和管理员提供手工失效、批量刷新、排障查看入口。 |
| 审计联动 | 所有手工失效和治理动作进入统一审计。 |

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `HJO2OA-Config` | 配置模块可定义缓存参数，但缓存策略真相源仍由本模块维护。 |
| 上游 | `HJO2OA-EventBus` | 本模块消费领域事件触发缓存失效，不直接感知业务事务。 |
| 下游 | `01-07` 全业务域 | 业务模块使用统一缓存抽象，不允许自行跨域删除底层缓存。 |
| 下游 | `HJO2OA-Audit` | 手工失效、批量刷新、策略调整必须进入统一审计。 |
| 边界外 | 业务主数据 | 任何缓存命中结果都不能替代业务数据库、字典、配置等真相源。 |

## 核心能力

| 能力 | 输入 | 输出 | 关键约束 |
|------|------|------|----------|
| 策略管理 | `namespace`、`backendType`、`ttlSeconds`、`evictionPolicy` | `CachePolicy` | 命名空间全局唯一。 |
| 事件驱动失效 | 统一事件信封、失效映射规则 | 失效记录、广播结果 | 只能通过规则映射触发，不允许硬编码跨域删缓存。 |
| 手工治理 | 命名空间、Key 模式、原因 | 手工失效结果、审计记录 | 治理动作必须可追溯。 |
| 监控统计 | 命中、未命中、容量、热点 | 仪表盘与告警输入 | 统计结果是运行视图，不是业务数据。 |
| 排障回源 | 命中失败、失效失败、缓存污染线索 | 回源建议、治理记录 | 回源重建优先于手工篡改缓存值。 |

## 典型业务流

1. 管理员定义缓存命名空间，设置后端类型、TTL、淘汰策略和失效模式。
2. 业务模块通过统一缓存抽象写入和读取缓存值，缓存仅承担读性能优化。
3. 配置、字典、内容发布或门户模板变更等事件触发缓存策略匹配，本模块执行失效并记录 `CacheInvalidationRecord`。
4. 若出现热点、污染或人工修复需求，运维人员通过治理台执行手工失效，动作写入统一审计。
5. 下游若发现缓存恢复失败，应重新回源构建缓存值，而不是将临时值视为权威数据。

## 一期实现约束

- 一期优先支持 Redis 命名空间、基础 TTL、事件驱动失效和手工失效，不建设多级缓存依赖图。
- 一期监控仅提供基础命中率、容量和失效次数，不提供预测性热点分析。
- 一期不开放业务模块绕过策略直接操作底层缓存实例。
- 失效记录以追加写入为主，不对单条失效结果做人工覆盖修正。
- 手工治理入口必须带权限控制、二次确认和审计留痕。
