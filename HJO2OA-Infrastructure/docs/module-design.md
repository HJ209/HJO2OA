# 00-平台基础设施 模块设计

## 1. 文档目的

本文档定义 `00-平台基础设施` 模块的整体设计策略、分层架构、核心职责边界、跨模块集成方式和一期交付范围。作为模块功能文档、领域模型文档和开发任务文档的上层约束文档。

对应架构决策编号：D02（模块化单体 + 独立基础设施服务）、D06（租户与组织分离）、D11（统一接口契约）、D12（统一事件契约）、D13（最终一致性）、D20（一期实施范围）、ADR-006（参考设计策略）。

## 2. 设计策略

### 2.1 参考设计原则

本模块参考 O2OA 及通用企业平台的基础能力建设思路，但不照搬其产品结构与技术堆叠。具体原则：

- **借鉴平台底座闭环**：事件驱动、统一附件、统一字典、统一配置、统一审计、统一错误码等基础能力闭环直接借鉴成熟思路。
- **不借鉴业务侵入式底座**：共享能力必须保持“被业务引用”，不能反向吞并业务语义主模型。
- **不借鉴大而全平台中心化实现**：HJO2OA 一期仍以模块化单体为主，基础设施能力优先做成统一组件和统一边界，不提前拆为全面微服务。
- **优先统一契约，再统一实现**：先收敛事件契约、错误码契约、配置覆盖链、附件绑定协议，再决定具体存储和中间件实现。

### 2.2 核心边界原则

| 原则 | 说明 |
|------|------|
| 共享底座拥有技术真相源，不拥有业务真相源 | 事件信封、配置策略、附件元数据、字典、审计归 `00`；业务事实归业务域 |
| 租户隔离不承担组织授权语义 | 租户负责系统隔离，组织/岗位/角色仍归 `01` |
| 事件总线只管契约和可靠性 | 事件语义、业务补偿和业务幂等仍由业务模块承担 |
| 缓存是性能手段，不是真相源 | 缓存策略可建模，缓存值不可取代主数据 |
| 平台工具必须可审计、可配置、可回滚 | 配置、安全、租户、关键资源变更都必须可追溯 |
| 共享资源优先抽象为统一能力 | 业务模块不得自建附件中心、字典系统、错误码体系、配置中心平替 |

### 2.3 O2OA 能力借鉴清单

| O2OA/通用平台能力 | HJO2OA 一期保留 | HJO2OA 一期简化 | HJO2OA 暂缓 |
|-------------------|----------------|----------------|-------------|
| 统一事件总线 | ✅ | - | - |
| 统一附件中心 | ✅ | - | - |
| 统一字典与配置 | ✅ | - | - |
| 统一错误码与国际化 | ✅ | - | - |
| 统一审计留痕 | ✅ | - | - |
| 多语言数据支持 | ✅ | 基础翻译工作台 | - |
| 时区统一策略 | ✅ | - | - |
| 缓存治理 | - | 基础命名空间与 TTL | 高级依赖图/热点分析 |
| 定时任务平台 | - | 基础 Cron + 手工触发 | 复杂 DAG 编排 |
| 多租户隔离 | - | 共享库 + 租户字段优先 | 自动迁移独立库 |
| 安全工具集 | - | 基础密钥/脱敏/签名/密码策略 | 复杂自适应风控 |

## 3. 分层架构

`00` 不适合套用纯业务域的“设计时/运行时/读模型”结构，而更适合分为：**契约与策略层、共享资源层、执行与运行层、运维与治理层**。

### 3.1 契约与策略层（Contract & Policy Layer）

负责统一定义共享能力的规则、注册表、覆盖链和标准化协议。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `event-bus/` | 事件定义、订阅关系、事件契约注册 | `infra_event_definition`、`infra_event_subscription` |
| `i18n/` | 语言列表、资源包、回退策略 | `infra_locale_bundle`、`infra_locale_resource_entry` |
| `timezone/` | 系统/租户/用户时区策略 | `infra_timezone_setting` |
| `dictionary/` | 字典类型、字典项、级联规则 | `infra_dictionary_type`、`infra_dictionary_item` |
| `config/` | 系统参数、覆盖链、Feature Flag | `infra_config_entry`、`infra_config_override`、`infra_feature_rule` |
| `error-code/` | 错误码、严重级别、HTTP 状态、国际化键 | `infra_error_code_def` |
| `security/` | 密钥、脱敏、密码、签名、限频策略 | `infra_security_policy`、`infra_secret_key_material`、`infra_masking_rule`、`infra_rate_limit_rule` |
| `tenant/` | 租户、套餐、配额、初始化规则 | `infra_tenant_profile`、`infra_tenant_quota` |

**契约与策略层规则**：

- 所有共享能力都必须先有统一定义，再有具体运行实例。
- 策略对象优先描述“规则”和“范围”，而不是直接绑定某个具体中间件实现。
- 主体级覆盖（组织、角色、用户）只引用 `01` 输出的主体 ID，不拥有主体主数据。
- 租户策略与组织策略必须分离，避免租户模型侵入业务授权体系。

### 3.2 共享资源层（Shared Resource Layer）

负责平台级共享资源主数据和可被业务域复用的元数据对象。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `attachment/` | 文件元数据、版本、绑定关系、预览状态、配额占用 | `infra_attachment_asset`、`infra_attachment_version`、`infra_attachment_binding` |
| `data-i18n/` | 多语言翻译项、翻译状态、导入导出 | `infra_translation_entry` |

**共享资源层规则**：

- 共享资源主模型归 `00` 所有，但业务可见性与业务生命周期仍由业务域控制。
- 附件和翻译都采用“主资源 + 业务绑定/业务引用”的模式，不允许共享资源直接拥有业务对象生命周期。
- 资源元数据和实际存储解耦：例如文件二进制内容走对象存储，翻译展示走运行时解析。

### 3.3 执行与运行层（Execution & Runtime Layer）

负责共享能力的运行态对象、执行日志和可靠性保障。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `event-bus/` | 事件信封、路由、投递尝试、死信处理 | `infra_event_message`、`infra_event_delivery_attempt` |
| `scheduler/` | 任务定义、执行记录、失败重试、依赖触发 | `infra_scheduled_job`、`infra_job_execution_record` |
| `cache/` | 缓存命名空间、TTL、失效记录、指标采集 | `infra_cache_policy`、`infra_cache_invalidation` |
| `audit/` | 平台级审计记录、字段差异、归档 | `infra_audit_record`、`infra_audit_field_change` |

**执行与运行层规则**：

- 运行态对象与策略对象分离，避免执行失败污染定义真相源。
- 事件、任务、缓存失效和审计都应支持幂等、重试和追踪字段。
- 运行态对象以 append-only 或状态追加优先，避免覆盖式修改造成可追溯性丢失。
- 一切人工干预动作（重试、回放、手动失效、回滚）都应进入统一审计。

### 3.4 运维与治理层（Ops & Governance Layer）

负责平台能力的运营控制台、监控、告警、回放和人工干预入口。

| 子模块 | 核心职责 | 持久化 |
|--------|----------|--------|
| `event-bus/` | 订阅管理、死信处理、事件回放、异常排查 | 基于运行态表查询 |
| `audit/` | 审计查询、归档查询、导出 | 基于审计表查询 |
| `cache/` | 手工失效、命中率分析、热点 Key 观察 | 基于缓存策略与监控指标 |
| `config/` | 配置回滚、Feature Flag 灰度操作 | 基于配置表与审计表 |
| `tenant/` | 租户初始化、配额监控、状态管控 | 基于租户表查询 |
| `security/` | 安全告警、策略调整、异常行为审计 | 基于安全策略与审计表 |

**运维与治理层规则**：

- 运维层不能绕过领域规则直接修改底层数据；一切操作必须经过领域服务。
- 回放、回滚和手工修复属于治理动作，应具备权限控制、审计记录和二次确认。
- 监控与告警应优先围绕共享能力本身，而不是直接替代业务监控中心。

## 6. 身份上下文与租户隔离

### 6.1 租户隔离原则（D06）

- `tenantId` 是系统隔离维度，不是组织管理维度。
- 共享资源、配置、附件、审计、缓存策略均应支持全局或租户级边界。
- 是否支持组织/角色/用户级策略，不意味着这些主体归 `00` 所有，只是引用其 ID。

### 6.2 操作者与身份上下文引用

- 审计、事件、任务执行、安全异常等运行对象可以记录 `operatorPersonId`、`operatorAccountId`、`assignmentId` 等引用字段。
- 这些字段只用于追踪和隔离，不把身份上下文主模型复制进 `00`。
- 任何需要重新解析组织、角色、岗位语义的地方，都应通过 `01` 的统一接口完成。

## 7. 运行治理策略

### 7.1 事件治理

- 发布前校验事件定义和载荷结构。
- 发布后进入统一事件信封与投递生命周期。
- 失败进入重试，超限进入死信，死信可治理回放。
- 回放只重驱消费，不重演发布业务事务。

### 7.2 配置治理

- 配置解析遵循默认值、覆盖链、Feature Rule 的顺序。
- 热更新只允许作用于声明可运行时变更的配置。
- 关键配置操作必须可回滚，并写入统一审计。

### 7.3 资源治理

- 附件采用“元数据 + 对象存储 + 业务绑定”的统一模型。
- 字典更新、语言资源更新、多语言翻译更新应可触发缓存刷新。
- 安全策略与租户策略更新必须支持灰度、生效边界和审计追踪。

### 7.4 运维治理

- 手工重试、回放、失效缓存、恢复配置、初始化租户都属于治理动作。
- 治理动作必须具备权限控制、操作审计和可回滚性。
- 一期优先建设最小可用治理台，不追求完整 AIOps 能力。

## 9. 一期交付范围

### 9.1 一期包含

- `event-bus`：事件定义注册、发布、订阅、可靠投递、死信、事件审计
- `i18n`：语言管理、资源包、后端消息国际化、热更新
- `timezone`：统一 UTC 存储与时区转换策略
- `data-i18n`：基础多语言翻译值存储与查询
- `attachment`：上传、下载、预览、版本、业务绑定、配额统计
- `dictionary`：字典类型、字典项、缓存刷新、引用协议
- `config`：配置项、覆盖链、Feature Flag、热更新、回滚
- `audit`：平台级审计记录、字段差异、查询与归档
- `error-code`：统一错误码、消息键、前后端错误标准

### 9.2 一期简化

- `cache`：先支持 Redis 命名空间、TTL、事件驱动失效和基础监控
- `scheduler`：先支持基础 Cron、失败重试、手工触发，不做复杂 DAG 图形编排
- `tenant`：先以共享库 + 租户字段隔离为主，不做自动独立库迁移
- `security`：先支持基础密钥引用、脱敏、密码策略、签名与限频，不做复杂自适应风控

### 9.3 一期暂缓

- 复杂事件治理控制台与精细化回放策略
- 多级缓存依赖图和热点预测
- 自动租户迁移与高级套餐编排
- 高级安全异常画像与智能风控
- 全量统一任务 DAG 编排工作台

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| `00` 逐渐膨胀为“平台万能模块” | 强制共享能力只提供底座，不拥有业务语义主数据 |
| 事件总线变成隐式业务流程引擎 | 事件总线只提供路由和可靠性，业务补偿仍归业务域 |
| 租户模型侵入组织权限体系 | 明确租户与组织分离，只引用主体 ID |
| 共享能力更新影响面过大 | 配置灰度、Feature Flag、审计和回滚共同保障 |
| 缓存和附件被业务绕过 | 统一协议、统一网关、统一 SDK，禁止模块内重复实现 |
| 运维动作不可追溯 | 所有治理动作进入统一审计与权限控制 |
