# 错误码体系概览

## 模块定位

| 项 | 说明 |
|----|------|
| 模块角色 | `HJO2OA-ErrorCode` 是 `00-平台基础设施` 契约与策略层子模块，负责统一错误码定义、语义分组、HTTP 映射和国际化消息键。 |
| 策略对象 | `ErrorCodeDefinition` 负责定义错误码、严重级别、HTTP 状态、消息键和可重试属性。 |
| 运行对象 | 具体异常实例、调用栈、告警事件、前端错误弹窗和接口返回体属于运行对象。 |
| 边界强调 | 本模块是统一定义中心，不是业务异常真相源；业务异常产生、抛出和处理仍归各业务模块。 |

## 核心职责

| 职责 | 说明 |
|------|------|
| 错误码注册 | 维护错误码、所属模块、分类、严重级别和 HTTP 状态映射。 |
| 国际化绑定 | 维护 `messageKey` 与资源包的关联，为前后端输出统一文案键。 |
| 兼容治理 | 支持错误码废弃、兼容期管理和替代建议。 |
| 文档输出 | 面向开发、测试、前端和开放接口输出统一错误码目录。 |
| 语义约束 | 约束可重试属性和严重级别，供事件投递、调度和接口网关联动。 |

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `HJO2OA-I18n` | 只引用资源包和消息键，不维护语言资源正文。 |
| 上游 | `01-07` 全业务域 | 业务模块申请或引用错误码定义，但异常对象和处理逻辑仍在业务域。 |
| 下游 | 前端、开放接口、SDK | 统一消费错误码目录、HTTP 映射和消息键。 |
| 下游 | `HJO2OA-Audit`、监控告警 | 可用错误码做聚类和排障，但不拥有异常栈。 |
| 边界外 | 运行时异常事实 | 业务模块的异常上下文、调用堆栈和补偿决策不由本模块持有。 |

## 核心能力

| 能力 | 输入 | 输出 | 关键约束 |
|------|------|------|----------|
| 定义管理 | `code`、`moduleCode`、`category`、`severity`、`httpStatus` | `ErrorCodeDefinition` | `code` 全局唯一。 |
| 国际化绑定 | `messageKey`、语言资源包 | 面向前后端的统一消息键 | 不直接存业务语言正文。 |
| 文档查询 | 模块、分类、严重级别、废弃状态 | 错误码目录、文档页、导出结果 | 只读输出，不承载异常处理。 |
| 废弃治理 | 废弃标记、替代错误码、兼容期 | 新旧码兼容关系 | 已对外发布的错误码不得随意改语义。 |
| 可重试约束 | `retryable`、`severity` | 调度、事件投递和接口重试参考 | 仅提供策略属性，不直接执行重试。 |

## 典型业务流

1. 基础设施或业务模块为新场景申请错误码，声明所属模块、分类、严重级别、HTTP 状态和消息键。
2. 错误码模块完成语义审核并落库为 `ErrorCodeDefinition`，必要时绑定替代码和兼容期。
3. 业务模块在抛出异常或构造统一响应时引用该错误码定义，不在本地另建平行编号体系。
4. 前端、开放接口和文档页通过查询接口获取错误码目录与消息键，结合语言资源展示文案。
5. 错误码被废弃或调整元数据后，本模块发布 `infra.error-code.updated`，通知下游刷新目录和缓存，但不修改已发生的业务异常事实。

## 一期实现约束

- 一期优先支持人工维护的错误码目录、文档查询和前后端统一映射，不建设自动异常分类引擎。
- 一期统计能力仅面向错误码使用频次和引用关系，不替代观测平台的异常聚合。
- 一期通过 `messageKey` 绑定国际化资源，不在错误码模块内存储多语言正文。
- 一旦错误码对外发布，不允许无兼容期地重定义语义；废弃只能通过治理流程执行。
- 本模块不拦截业务异常，也不保存业务异常栈，仅定义统一编号和语义约束。
