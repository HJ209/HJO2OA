# 审计日志概览

## 模块定位

| 项 | 说明 |
|----|------|
| 模块角色 | `HJO2OA-Audit` 是 `00-平台基础设施` 执行与运行层 + 运维治理层子模块，负责平台级统一留痕、差异追溯、归档和导出。 |
| 策略对象 | 审计采集规则、脱敏展示规则、归档策略属于治理策略对象。 |
| 运行对象 | `AuditRecord`、`AuditFieldChange` 是 append-only 运行对象，用于记录已经发生的事实。 |
| 边界强调 | 本模块只做留痕，不做业务审批状态机，不替代流程轨迹、业务审批意见或运营工单系统。 |

## 核心职责

| 职责 | 说明 |
|------|------|
| 统一采集 | 接收 API 调用、事件消费、配置治理、缓存治理、手工运维等来源的审计写入。 |
| 差异追溯 | 记录对象维度、操作类型、操作者、时间点与字段前后值差异。 |
| 合规查询 | 提供面向审计员、合规人员和运维人员的筛选、查看、导出能力。 |
| 归档治理 | 将过期审计记录归档，并在归档后保留可检索索引与归档审计事件。 |
| 脱敏展示 | 对敏感字段提供脱敏后的查询展示，原值访问受更高权限约束。 |

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `01-07` 全业务域 | 业务模块输出审计写入命令或事件，本模块统一留痕；业务模块自己的轨迹日志可并存。 |
| 上游 | `HJO2OA-Config`、`HJO2OA-Cache`、`HJO2OA-Tenant`、`HJO2OA-Security` | 所有治理动作必须进入统一审计，但这些模块各自维护自身策略真相源。 |
| 下游 | 合规、风控、运维 | 下游只查询和导出审计记录，不反向改写审计事实。 |
| 下游 | `HJO2OA-ErrorCode`、监控告警 | 可关联错误码、链路追踪和告警上下文进行排障，但不拥有这些对象。 |
| 边界外 | 业务状态迁移 | 审计记录不能驱动业务状态回退；任何“修复”都应回到原业务模块执行。 |

## 核心能力

| 能力 | 输入 | 输出 | 关键约束 |
|------|------|------|----------|
| 审计写入 | 对象标识、动作类型、操作者、差异详情 | `AuditRecord`、`AuditFieldChange` | 一旦写入不得修改，只允许归档。 |
| 多维查询 | 模块、对象、操作者、时间、租户、`traceId` | 审计列表、详情页 | 查询可筛选，结果只读。 |
| 敏感字段展示 | 差异值、敏感级别、访问权限 | 脱敏后的审计详情 | 默认脱敏，原值需单独授权。 |
| 归档与导出 | 时间范围、归档策略、导出条件 | 归档结果、导出文件、事件通知 | 归档后仍保留可检索索引和合规留痕。 |
| 事件联动留痕 | 统一事件信封、处理结果 | 跨模块行为追踪链 | 只记录事实，不接管事件补偿逻辑。 |

## 典型业务流

1. 业务模块或基础设施模块完成写操作、治理动作或事件消费后，向审计模块提交审计写入命令。
2. 审计模块追加 `AuditRecord`，并按需追加多个 `AuditFieldChange` 记录字段差异。
3. 管理员或合规人员按模块、对象、操作者、租户、时间范围查询审计记录。
4. 对敏感字段，前端默认展示脱敏值；如需查看原值，必须经过更高权限控制。
5. 达到归档阈值后，审计模块执行归档任务并发布 `infra.audit.archived`，供治理台和下游归档仓感知。

## 一期实现约束

- 一期优先支持统一查询、详情查看、归档和导出，不建设独立审计规则编排引擎。
- 一期以结构化字段查询为主，不提供完整 SIEM 或全文检索平台能力。
- 审计模块只保存事实记录和字段差异，不接管业务审批意见、会签轨迹或流程状态机。
- 一期归档以时间分层和模块分层为主，不支持复杂冷热分级策略。
- 所有手工修复、回放、回滚、失效缓存等治理动作都必须通过来源模块执行后再写入审计，禁止直接改审计记录。
