# tenant 概览

## 模块定位

`HJO2OA-Tenant` 是 `00-平台基础设施` 中负责系统隔离与租户治理的子模块，维护租户档案、套餐与配额、初始化编排和状态管控，为其他模块提供统一的租户边界。该模块只处理“系统租户”这一隔离维度，不承担组织树、岗位树、角色或账号主模型语义。

| 维度 | 说明 |
|------|------|
| 领域归属 | 契约与策略层 + 运维与治理层 |
| 真相源 | `TenantProfile`、`TenantQuota` |
| 核心目标 | 租户隔离、套餐与配额、初始化编排、状态治理 |
| 一期定位 | 共享库 + 租户字段隔离、初始化编排、配额预警 |
| 明确不负责 | 组织权限语义、账号主模型、租户内业务资源主数据 |

## 核心职责

| 职责 | 说明 | 关键输出 |
|------|------|----------|
| 租户档案管理 | 维护租户编码、名称、状态、默认语言/时区、隔离模式和套餐 | `TenantProfile` |
| 配额与套餐治理 | 维护存储、API、任务等配额上限与预警阈值 | `TenantQuota` |
| 初始化编排 | 编排默认配置、字典、门户、流程模板等租户初始化步骤 | 初始化批次与状态 |
| 状态管控 | 管理启用、停用、初始化中、初始化失败等治理状态 | 可审计的状态快照 |
| 运维入口 | 提供禁用、重试初始化、配额调整、回看审计等治理入口 | 治理操作记录 |

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `config` | 提供默认套餐、初始化模板和阈值规则；tenant 不拥有这些配置真相源。 |
| 上游 | `security` | 提供租户初始化所需的基线安全策略；tenant 不定义密钥、限频等安全细节。 |
| 上游 | `scheduler` | 可承载初始化补齐、配额对账等异步任务执行；tenant 拥有初始化语义，scheduler 只负责执行。 |
| 下游 | `attachment`、`dictionary`、`config`、`portal`、`process` 等模块 | 通过 `infra.tenant.*` 事件感知租户创建、初始化完成、停用和配额预警。 |
| 下游 | 运维治理台 | 展示租户状态、初始化轨迹、配额预警和变更审计。 |
| 边界外 | 组织树、岗位、角色、账号 | 这些主模型属于 `01-组织与权限`，tenant 只引用租户上下文，不拥有权限语义。 |
| 边界外 | 自动独立库迁移 | 一期不做自动独立库迁移与复杂跨库搬迁。 |

## 核心能力

| 能力 | 输入 | 输出 | 约束 |
|------|------|------|------|
| 租户档案 | 租户编码、名称、默认语言、默认时区、套餐 | 生效中的租户档案 | 租户编码唯一，状态变更必须留痕。 |
| 隔离策略 | 隔离模式、租户上下文、启停状态 | 隔离边界配置 | 一期以共享库 + 租户字段隔离为主。 |
| 配额治理 | 配额上限、预警阈值、当前用量 | 配额快照与预警结果 | 预警不等于自动删除资源，禁止越权清理业务数据。 |
| 初始化编排 | 初始化模板、步骤清单、目标模块 | 初始化状态、失败重试入口 | 初始化允许最终一致，但必须可重试、可审计。 |
| 状态治理 | 停用原因、恢复窗口、操作人 | 租户状态变更记录 | 高风险动作必须二次确认，不允许无审计停用租户。 |

## 典型业务流

1. 平台管理员创建租户，填写 `tenantCode`、名称、默认语言/时区、套餐和隔离模式。
2. tenant 校验编码唯一性与套餐规则后保存租户档案和基础配额，发布 `infra.tenant.created`。
3. 初始化中心按模板编排基础配置、字典、权限基线和默认资源的租户级初始化，必要时通过 scheduler 承载异步步骤。
4. 所有关键步骤完成后，tenant 将租户标记为可用并发布 `infra.tenant.initialized`；若步骤失败，保留失败原因和人工重试入口。
5. 运行期持续监控配额使用量，到达阈值时发布 `infra.tenant.quota-warning`；必要时平台管理员经二次确认后停用租户并发布 `infra.tenant.disabled`。

## 一期实现约束

| 约束 | 说明 |
|------|------|
| 隔离模式 | 一期仅保证共享库 + 租户字段隔离，不做自动独立库迁移。 |
| 权限边界 | 租户只负责系统隔离与配额，不承接组织、岗位、角色、账号的授权语义。 |
| 初始化方式 | 初始化采用编排 + 最终一致方式，不要求跨所有模块单事务完成，但必须保留失败重试入口。 |
| 配额策略 | 一期以预警和准入阻断为主，不直接自动删除业务资源。 |
| 治理要求 | 停用、初始化重试、配额调整等动作必须支持二次确认、审计与回滚或重试入口。 |
