# EventBus 模块概览

## 模块定位

| 项 | 说明 |
|----|------|
| 模块名称 | `HJO2OA-EventBus` |
| 模块定位 | `00-平台基础设施` 的事件契约与可靠投递中枢，统一管理事件定义、订阅关系、事件信封、投递尝试和死信治理 |
| 真相源边界 | 拥有 `EventDefinition`、`SubscriptionBinding`、`EventMessage`、`DeliveryAttempt` 等技术真相源；不拥有业务事件语义、不承担业务补偿决策、不替代业务流程编排 |
| 一期目标 | 建立统一事件注册、Outbox 出站、至少一次投递、死信治理和基础运维台，支撑 `01-07` 跨模块联动 |

## 核心职责

| 职责 | 说明 | 一期范围 |
|------|------|----------|
| 事件契约注册 | 管理事件类型、版本、载荷结构、发布方与订阅边界 | 支持注册、启停、兼容窗口说明和废弃标记 |
| 订阅关系管理 | 维护消费者、匹配方式、重试策略和死信策略 | 支持精确匹配与受控通配，不支持任意脚本路由 |
| 可靠出站 | 接收业务模块通过 Outbox 输出的事件，生成 `EventMessage` 并入总线 | 支持单条、批量、事务后发布 |
| 投递与死信治理 | 记录 `DeliveryAttempt`，处理重试、死信、回放和人工干预 | 支持至少一次投递、失败重试、死信入队 |
| 运行审计 | 汇总发布、投递、失败、回放等治理动作 | 支持运维查询、异常定位和统一审计留痕 |

## 上下游边界

| 方向 | 对象 | 边界说明 |
|------|------|----------|
| 上游 | `01-07` 各业务域与 `00` 其他基础设施模块 | 只接收已注册的跨模块事件信封，不解释业务语义 |
| 上游 | `HJO2OA-Config`、`HJO2OA-Security` | 复用配置、签名、限流和安全基线，不在总线内自建策略中心 |
| 下游 | `HJO2OA-Audit`、`HJO2OA-Cache`、`06` 消息与触达、`07` 治理与集成 | 向下游投递事件或输出运行状态，不直接改写下游真相源 |
| 非本模块 | 业务补偿与业务幂等 | 重复消费幂等、业务补偿、业务回滚由消费方承担 |
| 非本模块 | 业务流程编排 | 事件总线提供契约与可靠投递，不承担 Saga 或工作流编排语义 |

## 核心能力

| 能力 | 关键对象 | 输出 | 一期不包含 |
|------|----------|------|------------|
| 契约目录 | `EventDefinition` | 统一事件类型、版本和载荷门禁 | 任意临时事件直发 |
| 订阅绑定 | `SubscriptionBinding` | 消费清单、路由匹配和投递策略 | 非受控广播和脚本化转发 |
| 事件信封 | `EventMessage` | 平台级统一事件信封与出站记录 | 业务事件语义解释 |
| 投递尝试 | `DeliveryAttempt` | 重试、死信和失败轨迹 | Exactly-once 保证 |
| 回放治理 | 死信与人工操作记录 | 运维回放、异常定位和治理入口 | 绕过审计的手工改库补发 |

## 典型业务流

| 步骤 | 流程说明 | 关键产出 |
|------|----------|----------|
| 1 | 业务模块在本地事务内写入业务数据和 Outbox 记录，并引用已注册事件定义 | 待发布事件 |
| 2 | EventBus 校验 `eventType`、`eventVersion`、载荷结构和订阅关系，生成 `EventMessage` | 已入总线事件信封 |
| 3 | 总线按 `SubscriptionBinding` 对各消费者发起投递，逐次记录 `DeliveryAttempt` | 投递结果轨迹 |
| 4 | 投递失败进入重试窗口，超过阈值后进入死信并发布治理事件 | 死信记录与告警输入 |
| 5 | 运维台对死信执行回放或关闭，治理动作进入统一审计 | 回放记录与审计留痕 |

## 一期实现约束

| 约束 | 说明 |
|------|------|
| `00` 只拥有技术真相源 | 事件总线只拥有事件契约和投递运行态，不拥有任何业务事实真相源 |
| 策略对象与运行对象分离 | `EventDefinition`、`SubscriptionBinding` 是策略对象；`EventMessage`、`DeliveryAttempt` 是运行对象，禁止混写污染 |
| 发布必须经 Outbox | 不允许业务模块绕过事务边界直接手工补发未入账事件 |
| 投递语义为至少一次 | 总线只保证至少一次；消费者必须自行保证幂等与补偿 |
| 回放不重演业务事务 | 死信回放只重驱动投递，不重新执行发布方业务写操作 |
| 治理动作必须审计 | 重试、回放、关闭死信、停用订阅等动作都必须带操作人、原因和审计记录 |
